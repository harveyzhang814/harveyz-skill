# 研究方向：Skill 运行时文件的存储位置

> 本文件在澄清对话结束后立刻写入，是整个研究的起点。研究完成前不修改。

## 研究命题

Skill 在运行过程中产生的配置文件和过程文件（运行时文件），应当遵循统一的存储位置约定。

## 问题意识

目前缺乏约定，导致三类并发问题：
1. **文件散落**：不同 skill 各自存到不同路径，用户不知道去哪里找自己的数据，skill 之间可能相互干扰
2. **分类不清**：配置文件、缓存、brain/memory、session 状态等混放在一起，不清楚哪类应放哪里
3. **同步不明确**：不清楚哪些文件应跟着机器走（本地）、哪些应跟着项目走（可共享）

## 研究范围

**在范围内**：Skill 运行时产生的文件——
- 配置文件（用户对 skill 的偏好设置）
- 缓存文件（加速后续运行的中间结果）
- Brain/Memory 文件（跨 session 的知识积累）
- Session 状态文件（当前执行过程的中间状态）

**不在范围内**：
- SKILL.md 定义文件本身（安装约定已确定）
- Skill 输出的产物（报告、笔记、设计文档——路径由具体任务决定）

## 核心设计疑问

1. **全局目录 vs 项目目录**：运行时文件应存在 `~/.hskill/<skill-name>/`（全局、跨项目），还是 `./.hskill/`（项目内、可感知上下文）？还是两者分别存放不同类型？
2. **文件分类和层级结构**：config、cache、brain、session 这几种类型，是否需要在目录层级上做区分？如何组织才能既清晰又不过度设计？

## 研究目标

理解现有 Skill 生态在运行时文件存储上的不同实践——不是选一种方案，而是搞清楚各种方式背后的设计逻辑，以及它们分别适合什么情况。
