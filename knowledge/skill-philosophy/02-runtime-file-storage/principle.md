# 哲学二：Skill 运行时文件的存储位置

## 核心命题

Skill 在运行过程中产生的文件（配置、缓存、状态、memory）应遵循统一的路径约定，让用户能预测文件在哪里，让 skill 之间不相互干扰，让"跟机器走"和"跟项目走"的文件有明确区分。

**问题意识的来源**：没有约定时，每个 skill 自己决定存哪里，造成三类并发问题——文件散落难找、同类文件命名规则不一致、不清楚哪些文件应该进 git 或跨机器同步。

---

## 悬而未决的设计问题

### 问题一：全局目录还是项目目录？

运行时文件应存在全局用户目录（如 `~/.hskill/<skill-name>/`），还是项目本地目录（如 `./.hskill/`）？

- **全局目录**：配置和状态跟着用户走，跨项目复用；但项目无法感知（换项目时 skill 不知道切换了上下文）。
- **项目目录**：文件跟项目走，自然感知上下文；但每个项目都要重新建立配置，且需要 gitignore 处理。
- **两者分开**：不同类型文件放不同位置（如 config 全局、session 项目级）；逻辑清晰，但约定更复杂。

张力：**复用性与通用性** vs **上下文感知**。

> **研究结论**：研究未收敛，三种实践并存，选择依据是文件的"归属性"：
> - 跟用户走的（config、analytics、通用 cache）→ 全局目录，按 slug 关联项目（G stack 模式）
> - 本身属于项目的（输出规格、设计文档）→ 项目目录内，带命名空间前缀（Superpowers 模式）
> - 有明确工作区语义的 skill → CWD 即工作区，用 cd 切换（MSkill teach 模式）
>
> G stack 发现了最细腻的解法：同一根目录下，按"是否跟特定项目绑定"再分层（全局子目录 vs `projects/$SLUG/` 子目录）。→ 详见 [[research]]

### 问题二：运行时文件是否需要按类型分目录？

config、cache、brain/memory、session 这几种文件是否需要在目录层级上做区分？

- **扁平结构**：所有文件直接放在 `<skill-name>/` 下，用文件名区分类型；简单，但量多时难以浏览。
- **按类型分子目录**：`<skill-name>/config/`、`<skill-name>/cache/` 等；结构清晰，但对于只有 1-2 个文件的 skill 过于繁重。

张力：**清晰度** vs **最小化约定**。

> **研究结论**：G stack 和 MSkill teach 都明确按关注点分子目录（checkpoints/、analytics/、reference/、lessons/），且按**关注点**而非时间分。Superpowers 不分子目录，但文件数量也少。结论：**按类型分子目录是普遍实践**，但当文件数量少时，命名规则可以替代子目录（三者都用文件名携带时间戳或序号等元数据）。→ 详见 [[research]]

## 尚未解答的更深问题

- 哪些运行时文件应该进 git，哪些永远不应该进 git？（尤其是 brain/memory 文件——它包含了用户数据，但也是 skill 跨 session 运作的核心）
- 如果 skill 同时需要"全局配置"和"项目级覆盖"，如何处理优先级和合并逻辑？
- Session 状态文件在任务完成后是否应该清理？清理策略是什么？

> **研究新增**：G stack 的"空 touch 文件作为功能标记"（`.proactive-prompted`、`.telemetry-prompted`）是一个没有在其他生态出现的独特模式——用文件存在与否表达 boolean 状态，而不是写入配置值。这个模式值得单独研究：什么时候用文件存在/不存在优于配置键值对？
