# 核心命题

在全栈开发中，同一业务概念在前端、后端、AI 对话中往往有不同叫法。Skill 应该维护一份跨越这些边界的领域术语表，作为所有参与方的收敛锚点——既不是技术层的命名规范，也不是一次性文档，而是整个开发过程中持续演化的"项目通用语"。

**解决的问题**：命名漂移导致的沟通摩擦——前后端叫法不一，AI 每次 session 自行发明术语，人类讨论时混用多种说法。

---

# 悬而未决的设计问题

## 问题一：触发时机——"遇到时更新"还是"专门时刻整理"？

**选项 A：实时触发**——每当在对话中遇到模糊/冲突术语，立即更新术语表。
*张力*：低摩擦，不会遗漏；但打断工作流，每次对话都可能触发文件写入。

**选项 B：专门触发**——用户主动调用术语整理 skill，从当前对话中提取并存档。
*张力*：不打扰工作流；但依赖用户记得主动触发，容易遗漏关键术语。

**研究结论**：MSkill 从"专门触发"（`ubiquitous-language`）演化为"实时触发"（`domain-modeling`），方向明确。实时维护是更成熟的策略，尤其在多 session 跨越多周的项目中。但实时触发需要 AI 有清晰的"何时更新"判断力——`domain-modeling` 用"术语冲突就立刻更新，不批量积累"解决了这个问题。

---

## 问题二：词汇归属——词汇表是项目的，还是工具的？

**选项 A：项目级**——术语表存在 repo 里（如 `CONTEXT.md`），属于这个项目。
*张力*：精准、可版本控制；但每个项目都要从零建立。

**选项 B：工具级**——术语表是 skill 生态的一部分，覆盖通用技术术语。
*张力*：拿来即用，无需维护；但无法覆盖项目特定的业务概念。

**研究结论**：两者服务于不同目的，不互斥。工具级（gstack `jargon-list.json`）解决的是"解释通用技术术语"的问题；项目级解决的是"统一业务领域概念"的问题。用户的场景（前后端业务命名对齐）属于后者，需要项目级术语表。

---

## 问题三：前后端冲突时谁说了算——统一到一个词，还是记录映射？

**选项 A：强制统一**——选一个规范词，其他叫法列为"Avoid"。
*张力*：干净彻底；但可能破坏已有代码里的命名，或造成前后端代码名称与术语表不一致。

**选项 B：记录映射**——保留两种叫法，明确标注"前端叫 X，后端叫 Y，指同一个概念"。
*张力*：务实，不破坏现有代码；但长期维护两套叫法增加认知负担。

**研究结论**：研究未完全收敛。MSkill `domain-modeling` 倾向强制统一（选一个规范词 + Avoid 列表），并用"例句对话"展示规范词在真实使用中的样子。gstack 也是强制统一（jargon-list 只有规范词，没有映射关系）。但两者均未讨论已有代码命名冲突的处理。实践建议：优先统一，但允许在术语条目里注明"代码中暂用 Y，目标统一为 X"作为过渡标记。

# 尚未解答的更深问题

1. **跨 session 持久性**：AI 在新 session 如何知道要读术语表？需要靠 CLAUDE.md 指令、还是每次 session 开头的 skill 调用？
2. **多人协作**：当多个开发者（或多个 AI agent）同时在项目中工作，谁负责更新术语表？如何避免冲突？
3. **演化管理**：当一个术语的定义发生实质变化时，历史代码和历史文档里的旧用法怎么处理？
