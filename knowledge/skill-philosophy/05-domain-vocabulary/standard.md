# 领域术语统一规范

> 适用于：需要跨前后端、跨 AI/人类对话维护统一业务术语的项目 Skill

---

## 核心原则

**术语表是活文档，不是一次性产物。** AI 在对话中遇到命名冲突时立即更新，不批量积累。术语表只存领域层概念，不存技术实现细节。

---

## 三种词汇管理策略

根据需求选择策略（可组合）：

| 策略 | 适用场景 | 词汇归属 | 触发方式 |
|------|----------|----------|----------|
| **主动守卫式** | 跨 session 项目，需要持久积累 | 项目级（`CONTEXT.md`） | 实时（遇到冲突即更新） |
| **计划级约束式** | 单次实现任务内的命名一致性 | 计划文档（Global Constraints） | 计划起草时声明 |
| **嵌入式词汇** | Skill 自身有专有概念需要固化 | Skill 定义本身 | 随 Skill 调用自动生效 |

**用户场景（前后端 + AI/人类跨边界对齐）首选：主动守卫式 + 计划级约束式组合。**

---

## 创建时：主动守卫式 Skill 的检查

1. **有 `CONTEXT.md` 指令吗？** — Skill 应指令 AI 在 session 开始时读取 `CONTEXT.md`
2. **冲突检测规则写清楚了吗？** — 明确说"遇到与词汇表不一致的用词，立即指出并更新"
3. **写入时机是实时还是批量？** — 应为实时；"下次整理"会遗漏关键上下文
4. **词汇表和实现细节是否混入？** — `CONTEXT.md` 只存业务领域概念，技术命名不进表
5. **有 Avoid 列表吗？** — 每个术语条目应标注哪些说法需要避免

---

## Review 时：识别问题的信号

| 信号 | 处理方式 |
|------|----------|
| AI 在同一 session 内用了多种叫法 | 检查是否有冲突检测指令；或 `CONTEXT.md` 未在 session 开始时读取 |
| 词汇表同时包含业务概念和函数名 | 拆分：业务概念留 `CONTEXT.md`，技术命名用 ADR 或代码约定管理 |
| 词汇表是在对话结束后一次性生成的 | 改为实时维护；或说明场景确实只需一次性提取（短会话） |
| 没有 Avoid 列表，只有定义 | 补充 `Avoid: X, Y`，使规范词和禁用词都明确 |
| 新 session 开始后 AI 不认识词汇表里的概念 | 在 CLAUDE.md 或 Skill preamble 加"读取 `CONTEXT.md`"指令 |
| 前后端的代码命名与词汇表出入很大 | 在词汇表条目里加"代码暂用 X，目标统一为 Y"过渡标记，不强制即时重命名 |

---

## 可直接复制的起始模板

**使用时需要替换的占位符：**
- `{项目名}` — 项目的领域名称（如"电商订单系统"）
- `{术语 N}` — 规范术语
- `{定义 N}` — 一两句话说清楚这个概念是什么（不是它做什么）
- `{避免用词 N}` — 该术语的同义词或曾经使用过的混乱叫法

**使用时需要删除的内容：**
- 不适用的策略变体
- `← 注释` 行

---

### 策略 A：主动守卫式（`CONTEXT.md` + Skill 指令）

**Step 1 — CLAUDE.md 加入全局指令：**

```markdown
## 领域术语

每次新 session 开始时，读取项目根目录的 `CONTEXT.md`。
遇到与 `CONTEXT.md` 不一致的用词时，立即指出："你刚才用了 X，词汇表里这个概念叫 Y。我现在更新 CONTEXT.md。"
然后更新文件，再继续当前任务。
```

**Step 2 — 创建 `CONTEXT.md`：**

```markdown
# {项目名}

{一两句话说明这个项目的业务边界和存在意义。}

## Language

**{术语 1}**:
{定义 1}
_Avoid_: {避免用词 1}

**{术语 2}**:
{定义 2}
_Avoid_: {避免用词 2}

**{术语 3}**:
{定义 3}
_Avoid_: {避免用词 3}
```

---

### 策略 B：计划级约束式（嵌入 `writing-plans`）

在每份实现计划的 `## Global Constraints` 段里加入：

```markdown
## Global Constraints

**术语约定（与 `CONTEXT.md` 一致）：**
- {领域概念 1} 统一叫 `{术语 1}`，不用 {避免用词 1}
- {领域概念 2} 统一叫 `{术语 2}`，不用 {避免用词 2}
- 函数名、API 字段名、组件名均遵循上述约定

**跨任务一致性检查：**
在提交计划前，确认所有任务的 Interfaces 签名中出现的术语与上述约定一致。
```

---

### 策略 C：嵌入式词汇（在 Skill 内部固化专有语言）

适用于 Skill 本身有一套固定的领域词汇（如 DDD 术语、设计模式概念）：

```markdown
## 词汇表

使用以下术语，不要用括号内的替代说法：

**{术语 1}** — {定义 1}。_Avoid_: {避免用词 1}

**{术语 2}** — {定义 2}。_Avoid_: {避免用词 2}

这套术语在本 Skill 范围内是固定的，不随项目变化。项目特定的业务词汇请参阅 `CONTEXT.md`。
```
