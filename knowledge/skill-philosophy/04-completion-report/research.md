# 研究报告：Skill 完成回报的设计哲学

> 关联文档：[[principle]]
> 示例场景：一个执行"文章抓取并翻译"任务的 Skill，需要在完成后向用户汇报

---

## 研究对象

| 生态 | Skill | 类型 |
|------|-------|------|
| G stack | `ship` — Completion Status Protocol | 执行类，部署工作流 |
| Superpowers | `finishing-a-development-branch` | 执行类，分支收尾工作流 |
| MSkill | `triage` | 执行类，Issue 状态机管理 |

---

## 阶段四：各研究对象解析

### 4.1 G stack — `ship` Completion Status Protocol

**核心机制——状态词汇表：**

```
DONE             — 完成，附证据
DONE_WITH_CONCERNS — 完成，但列出关切点
BLOCKED          — 无法推进，说明阻塞原因和已尝试的方法
NEEDS_CONTEXT    — 缺少信息，说明需要什么
```

**格式约定：** 四个字段，但只在非 DONE 状态时完整填写：
`STATUS` / `REASON` / `ATTEMPTED` / `RECOMMENDATION`

DONE 状态下，format 是 "DONE — completed with evidence"——状态标签加证据散文，没有固定字段结构。

**关键设计决策（明确写出的）：**
- 状态是预定义词汇，四选一
- 证据是散文（不是结构化字段）
- 失败状态（BLOCKED/NEEDS_CONTEXT）有额外字段（REASON、ATTEMPTED、RECOMMENDATION）
- 遥测（telemetry）是独立的机器可读日志，不是给用户看的

**关键设计决策（隐含的）：**
- **状态词汇 + 证据散文是分离的两层**：状态是机器/人都能秒读的标签；证据是给人细读的内容
- **成功状态（DONE）格式轻于失败状态**：失败时需要更多结构（ATTEMPTED、RECOMMENDATION）；成功时只需状态标签和证据
- **遥测与回报是不同的目的**：遥测是写给机器（jsonl 日志），回报是写给人

**原文关键句：**

> "DONE — completed with evidence."

解读：evidence 是必要的，但格式自由。"with evidence" 隐含了"不能只说完成，要说明做了什么"。

> "Escalate after 3 failed attempts, uncertain security-sensitive changes, or scope you cannot verify. Format: STATUS, REASON, ATTEMPTED, RECOMMENDATION."

解读：结构化字段只在需要升级时启用——平时散文，遇到复杂情况才上结构。

### 4.2 Superpowers — `finishing-a-development-branch`

**核心机制——动作即回报：**

这个 Skill 几乎没有显式的"完成回报"——完成的通信方式是**执行动作本身**。

各选项的"回报"：
| 选项 | 执行结果 | 显式回报 |
|------|----------|----------|
| Option 1：本地 Merge | merge 成功 + 测试通过 + cleanup | 无（执行成功说明一切） |
| Option 2：Push & PR | PR 创建成功 | PR 链接（GitHub 自动提供） |
| Option 3：保持现状 | 无动作 | `"Keeping branch <name>. Worktree preserved at <path>."` |
| Option 4：丢弃 | 确认后删除 | 无（执行成功说明一切） |

**关键设计决策（明确写出的）：**
- Option 3 有一句明确的回报——因为"无动作"本身无法证明"完成"
- Option 4 需要打字确认（"discard"）——确认动作即是用户已知晓的证据

**关键设计决策（隐含的）：**
- **不需要独立的完成报告，只有在"动作无法自证"时才输出文字**
- 回报是情境驱动的，不是格式驱动的
- 没有全局状态标签（无 DONE/FAILED 词汇）
- 成功和失败的区分来自流程控制（测试失败则 Stop），而不是来自回报格式

**原文关键句：**

> `Report: "Keeping branch <name>. Worktree preserved at <path>."`

解读：这是全文唯一一处明确说"Report:"的地方——恰好是"没有发生任何动作"的情况。说明这个设计里，"动作自证"是默认预设，只有动作缺席时才需要文字回报。

### 4.3 MSkill — `triage`

**核心机制——结果决定模板：**

不同的 triage 结果，产出完全不同的产物：

| 结果 | 产出形式 | 接收者 |
|------|----------|--------|
| `ready-for-agent` | Agent Brief 评论 | 下游 AI agent |
| `ready-for-human` | 同 Agent Brief 结构，但注明不可委托原因 | 人类开发者 |
| `needs-info` | Triage Notes 评论（模板：已确认/仍需要） | Issue 报告者 |
| `wontfix` | 关闭评论 + 写入 .out-of-scope/（如适用） | Issue 报告者 |

**关键设计决策（明确写出的）：**
- 模板有固定的"开始免责声明"：`> *This was generated by AI during triage.*`
- `needs-info` 模板结构固定（"已确认 / 仍需要"两段）
- 产物是写入外部系统（issue tracker），不是向调用者汇报

**关键设计决策（隐含的）：**
- **回报的对象是第三方（issue 系统），不是调用者**：triage 的"完成"意味着在外部系统留下痕迹
- **同一个动作（完成一次 triage）可以产生多种格式的产物**，由结果类型决定
- 没有向调用者（maintainer）的显式完成回报——maintainer 看到 AI 在 issue 上留言即知道完成了

**原文关键句：**

> "Every comment or issue posted to the issue tracker during triage must start with this disclaimer: `> *This was generated by AI during triage.*`"

解读：这是 MSkill 里最接近"完成回报格式"的设计——强制的免责声明，确保接收产物的人知道这是 AI 产出。

## 阶段五：核心思路提炼

### 思路 A：状态词汇 + 证据散文（G stack ship）

**核心逻辑：** 预定义 4 个状态词汇，强制 AI 用标签而不是散文来声明结果；证据和细节用自由散文补充。

**特征：**
- 状态是固定词汇（不能自创），秒读
- 成功状态简洁，失败状态有结构化字段（REASON、ATTEMPTED、RECOMMENDATION）
- 遥测与人读回报完全分离
- 不强制卡片壳，只强制状态标签

**适用场景：** 执行类 Skill，结果是二元的（成功/失败/阻塞/缺信息），调用者需要快速判断是否成功

**边界：** 批量任务时，4 个状态不足以表达"部分完成"的细粒度；状态词汇是英文，中文场景需翻译

**示例写法（文章抓取场景）：**
```
DONE — 已抓取并翻译，原文存至 Origin/article.md，译文存至 article.md
```

```
DONE_WITH_CONCERNS — 已抓取并翻译，但原文图片无法下载（3/5 张 403 错误）
- 产物：origin/article.md（含原图占位）、article.md（含图片路径标注）
- 关切：图片缺失可能影响阅读完整性
```

---

### 思路 B：动作自证，例外才输出（Superpowers）

**核心逻辑：** 成功的动作不需要报告——执行本身就是证明。只有在"没有动作发生"或"动作结果不自明"时，才用最少的文字说明。

**特征：**
- 无固定格式，无状态标签
- 成功 = 执行完成，结果可见（PR 链接、文件存在）
- 只有"无动作"场景（Option 3）才有一句明确的 Report 文字
- 失败通过流程控制体现（Stop，不执行后续），不通过回报格式体现

**适用场景：** 执行结果在外部系统可见的 Skill（PR、commit、文件）；用户本来就能在操作结果里直接看到是否成功

**边界：** 当 Skill 产物是隐藏在深层路径的文件时，"动作自证"失效——用户不知道去哪找；批量任务时无法汇总

**示例写法（文章抓取场景）：**
```
（无固定输出——若翻译成功，用户可在 Vault 根目录找到文件）

只有在跳过时才输出：
"已跳过（dedup 命中：同 URL 已存档于 20250601-article.md）"
```

---

### 思路 C：结果决定模板，产物面向第三方（MSkill triage）

**核心逻辑：** 不同结果类型对应完全不同的产物形式和接收者；"完成回报"是写给外部系统的，不是写给调用者的。

**特征：**
- 每种结果有自己的模板（Agent Brief / Triage Notes / 关闭理由）
- 产物是公开的（issue 评论、issue 状态变更），调用者通过看外部系统了解完成
- 强制的 AI 免责标头，接收者知道是 AI 产出
- 无向调用者的显式"任务完成"汇报

**适用场景：** Skill 的完成等于在外部系统留下可检索的产物（issue tracker、PR comment、文档）；调用者不是回报的主要接收者

**边界：** 如果 Skill 的产物不在外部系统里（只有本地文件），调用者无法通过"看外部系统"来确认完成；调用者自己也需要一个"完成"的明确信号

## 阶段六：对比与回答

### 对比表

| 维度 | 思路 A（状态词汇 + 散文） | 思路 B（动作自证） | 思路 C（结果决定模板） |
|------|--------------------------|-------------------|----------------------|
| **格式强度** | 中（状态强制，证据自由） | 弱（几乎无格式约定） | 强（每类结果有模板） |
| **状态词汇** | 预定义 4 个，强制使用 | 无 | 无（结果是外部系统状态） |
| **回报接收者** | 调用者 | 调用者（隐式） | 第三方（issue 系统） |
| **批量场景** | 需要扩展（4 状态不够） | 无汇总机制 | 每 item 独立产物 |
| **失败信息量** | 高（REASON + ATTEMPTED + RECOMMENDATION） | 通过 Stop 体现，无结构 | 取决于结果类型 |
| **可扫描性** | 高（状态标签秒读） | 低（需读动作上下文） | 中（需找到外部系统） |

### 回答 `principle.md` 中的悬而未决问题

#### 问题 1：强结构 vs. 弱结构

**研究结论：两级结构是最优解。**

G stack 给出了最清晰的答案：**状态标签强制（词汇表），证据格式自由（散文）**。两级结构分离了"用户需要秒读的部分"和"用户需要细读的部分"。

完全弱结构（Superpowers）在执行结果自明时体验最好，但在批量任务或产物不可见时失效。
完全强结构（per-outcome templates）会随结果类型增多而维护成本增高。

**建议**：状态层强制（固定词汇），证据层自由（散文或结构化字段视复杂度选择）。

---

#### 问题 2：状态词汇应该预定义，还是自由表达？

**研究结论：预定义，但要覆盖批量场景。**

G stack 的 4 个状态（DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT）缺少"部分完成"（PARTIAL）状态。在批量场景中需要补充。状态词汇的价值在于可扫描性——用户不需要读完一段话才知道成功了没有。

---

#### 问题 3：完成回报是对调用者的，还是应该产出面向相关方的产物？

**研究结论：两者分离，各自独立。**

MSkill triage 清楚地展示了两者的区别：面向调用者的回报（"我完成了 triage"）和面向相关方的产物（issue 评论）是两个不同的输出。当 Skill 需要产出面向第三方的产物时，这两者应该明确分开，不应该在"完成回报"里把两者混为一谈。

---

#### 新发现：批量回报的时机

Superpowers 没有批量机制，G stack 的状态是 per-skill 而非 per-item，MSkill triage 是每 item 独立评论。

**推论**：批量任务的最佳实践是"每项完成即输出一张"（MSkill 的隐含做法），而不是等全部完成再汇总——这样用户可以实时知道进度，不会因为等待全部完成而焦虑。

## 现有 standard.md 的评估

对照研究结论，`standard.md`（先行实践）的设计：

| 设计点 | standard.md 的做法 | 研究支撑度 |
|--------|-------------------|------------|
| 卡片壳结构 | 固定格式 `── {状态} ──` | ✓ 状态强制符合 G stack 原则 |
| 四种状态值 | 完成/失败/部分完成/已跳过 | ✓ 覆盖了 G stack 缺少的 PARTIAL |
| 字段自定义 | 各 Skill 自行定义字段 | ✓ 证据层自由，符合研究结论 |
| 批量模式 | 每项完成立即输出 | ✓ 与 MSkill 的隐含做法一致 |
| `原因` 字段规则 | 失败/部分/跳过时必须出现 | ✓ 对应 G stack 的 REASON 字段 |
| 遥测 | 未提及 | — 不在 Skill 回报范围，无需处理 |

**结论：standard.md 的核心设计是合理的，与研究结论高度一致。** 可以补充"引用规范时应说明哪些字段是必填的"这条细节。
