# 研究方向：Skill 完成回报的设计哲学

## 研究命题

Skill 执行完成后，如何向用户汇报结果？

## 问题意识

不同 Skill 的完成回报格式各异，有的是散文、有的是列表、有的无输出——导致用户不知道任务是否成功、产物在哪、为什么失败。需要一套统一的回报格式规范，让用户在任何 Skill 完成后都能快速读懂结果。

## 边界与例外

- **纯参考类 Skill** 不适用：只返回信息，无动作，没有"完成"这个概念
- **中间 Skill（子步骤）** 不适用：作为其他 Skill 的子调用，回报由调用方统一处理
- **大部分用户主动触发的执行类 Skill 都应该有完成回报**，包括执行结果、产物位置、失败原因

## 核心设计疑问

**强结构 vs. 弱结构**：完成回报应该是强结构（固定卡片壳、字段顺序统一）还是弱结构（约定必填字段但格式自由）？

两种选项的张力：
- 强结构：一致性高，用户可以习得扫读模式；但信息密度差异大的 Skill 套同一个壳会显得生硬
- 弱结构：适配灵活，每个 Skill 可以按自己的信息量决定格式；但跨 Skill 的阅读体验不统一

## 研究目标

理解各生态的取舍，找到执行类 Skill 在完成回报上的共识，更新现有 `standard.md`（已有初稿，研究完成后补充研究支撑）。

## 备注

本 direction 由现有 `standard.md` 逆向推导，再经用户确认边界后定稿。standard.md 为先行实践，研究为后补依据。
