# 命题归档

> 关联研究：[[research]]

## 核心命题

**Skill 中的代码块应只在模型无法自行推断时写入，否则用文字描述代替。**

这条原则解决两个问题：（1）Skill 作者低估模型推断能力，写入冗余代码块，增加上下文噪声；（2）过度约定压缩执行端 agent 的环境适应空间，在跨平台或工具版本差异的场景下反而提高出错概率。

---

## 悬而未决的设计问题

**选项 A：基于命令复杂度**
> 复杂 flag 组合、非标准参数顺序 → 保留；基础操作（mkdir、mv、cat）→ 去除

张力：复杂度是主观的。"复杂"对有经验的 agent 可能是常识，对新 agent 可能不是。

**选项 B：基于跨环境差异风险**
> 命令在不同平台/工具版本行为不同 → 去除（让 agent 自行适配）；行为跨环境一致 → 可保留

张力：去除反而让 agent 在不熟悉的环境下无从参考。

**选项 C：基于出错代价**
> 执行错了影响大（删数据、覆盖文件）→ 保留作参考；低风险操作 → 去除

张力：保留高风险命令代码块，和"让 agent 自己处理"原则相矛盾。

**研究结论：** 三个维度均有效，但不是并列的——它们是三层过滤。详见 [[research]] 「综合判断框架」。

- **语法层**（复杂度）：flag 组合/语法非显而易见 → 保留；标准操作 → 去除
- **环境层**（跨平台差异）：路径/工具名是生态特有的 → 保留；可通过 `--help` 自查 → 去除
- **脚手架层**（功能角色）：多步初始化的一部分 → 独立代码块；单条精确命令 → inline code；条件逻辑 → 散文

出错代价（风险性）不是独立维度——高风险操作的"危险"已通过语法层或功能层覆盖，不需要单独成为保留条件。

### 问题二：标准如何同时服务两种使用场景？

**场景 A：生成时遵守**（嵌入生成 Skill 的 agent）
> 标准要能被 agent 理解并内化，在生成 Skill 内容时主动遵守

**场景 B：审查时兜底**（人工 review 生成结果）
> 标准要清晰到人工能快速判断每个代码块是否应该保留

张力：生成 agent 需要的是原则（理解后自行判断）；人工审查需要的是 checklist（快速决策）。两种形式哪个优先，还是必须兼顾？

**研究结论：** 两种需求互补，不矛盾。原则用于生成时内化，checklist 用于审查时对照。推荐形式：「原则 + checklist」，生成 agent 读原则、人工 review 用 checklist 逐条验证。

### 问题三：灰色地带的边界案例

已知的明确保留案例：
- 非显而易见的语法（如 `git diff A...B --name-only`）
- 必须精确对齐的结构（JSON schema、输出格式模板）

已知的明确去除案例：
- 基础文件操作（mkdir、mv、find、ls、cat）

**尚不清楚的灰色地带：**
- 多步骤组合命令（每步简单，组合后不一定）
- 跨平台命令（macOS vs Linux 行为有细微差异）
- "容易出错"的普通命令（操作简单但历史上常搞错的）
- 带副作用的操作（curl、git push、rm -rf）

**研究结论（灰色地带 checklist）：** 见 [[research]]「回答悬而未决的设计问题 - 问题三」。核心新发现：
- 特定 CLI 工具路径（不可预测）→ inline code（不需要独立代码块）
- `--help` 可查的 flags → 去除，改写"运行 `--help` 查看"
- 多步骤初始化（mkdir + 写文件 + 配置）→ 保留为独立代码块（脚手架层）
- 带副作用操作 → 不依赖代码块表达风险，改用散文强调

## 尚未解答的更深问题

- 判断标准是否应该随 Skill 类型（执行类 vs 参考类）而变化？（研究倾向：应该——参考类几乎不需要代码块；执行类可用脚手架层判断）
- 如果一个命令对部分用户"显而易见"而对另一部分不是，如何决策？（研究未收敛）
- 这条原则的反面是什么？是否存在"代码块写得太少"的失效模式？（研究发现：MSkill 的激活式方法在纯执行类 Skill 中可能不足，但未量化）
