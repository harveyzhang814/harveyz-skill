# 研究方向

> 这是出发时的地图，研究完成前不应修改。

## 研究命题

Skill 中的代码块应只在模型无法自行推断时才写入，否则用文字描述代替。

## 问题意识

1. **低估模型能力** — Skill 作者（人或 agent）倾向于把显而易见的操作也写成代码块，造成上下文冗余，让 agent 做无效的信息处理。
2. **过度约定收窄调整空间** — 代码块隐含"照这个执行"的语义，约定越紧，执行端 agent 可根据环境自行调整的空间越小，出错概率越高。不同环境下（macOS vs Linux、工具版本差异），相同操作的命令形式可能不同，过度指定反而制造错误。
3. **Agent 自动生成场景** — 大部分 Skill 由 agent 生成，生成端不知道执行端的推断能力，天然倾向于过度约定，使问题在生成阶段就被放大。

## 边界与例外

- **只针对代码块**：示例输出、JSON schema、具体数值等其他 Skill 元素不在此次研究范围内。
- **模型能力假设固定**：不随模型版本调整判断标准，假设执行端具备合理的通用推断能力。

## 核心设计疑问

1. **判断元问题**：如何判断"模型能否自行推断"？命令复杂度、跨环境不确定性、操作风险性，各自权重如何？是否存在一个通用的判断框架？
2. **双端服务**：标准需同时服务于「生成时遵守」（嵌入生成 Skill 的 agent）和「审查时兜底」（人工 review 生成结果）。这两种使用场景对标准形式的要求是否一致？
3. **边界案例**：哪些情况是真正的灰色地带——感觉普通但实际容易出错，或者跨环境差异不明显但确实存在？

## 研究目标

产出一套**可操作的判断标准**，形式为：**原则为主 + checklist 补充边界案例**。

验收标准：
- 写 Skill 时遇到代码块决策不再犹豫
- 能回溯验证现有 Skill 中每个代码块该不该保留
