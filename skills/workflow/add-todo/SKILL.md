# 写入 TODO

将需求澄清、去重、打标签，并规范写入项目 TODO 文档。

## 阶段一 — 确定 TODO 文件路径

### 配置文件位置
TODO 文件路径按项目存储在 `.claude/todo-config.json`：

```json
{ "todo_path": "docs/TODO.md" }
```

路径相对于项目根目录（`.claude/` 所在目录）。

### 解析逻辑

**第一步 — 判断是否为重新配置请求。**
如果用户的意图是"换一个 TODO 文件"、"修改 todo 路径"等，直接跳到下方的**重新配置流程**。

**第二步 — 读取已保存的配置。**
检查 `.claude/todo-config.json` 是否存在，且 `todo_path` 指向的文件确实存在。若是，直接使用该路径，进入阶段二。

**第三步 — 首次使用（无配置或文件不存在）。**
若配置不存在或文件缺失：

1. 扫描项目根目录，按顺序查找：`TODO.md`、`BACKLOG.md`、`docs/TODO.md`、`docs/backlog.md`、`docs/BACKLOG.md`。
2. 若找到文件，询问用户："发现 `<路径>`，用这个作为 TODO 文件吗？还是指定其他路径？"
3. 若未找到，询问："未找到 TODO 文件，要在哪里创建？（默认：`TODO.md`）"
4. 若需要新建文件，使用以下初始结构：

```markdown
# TODO / Backlog

> 项目需求与待办事项。

## 🚧 待开发

<!-- 新增条目写在这里 -->

## ✅ 已完成

<!-- 已完成条目写在这里 -->
```

5. 将路径保存到 `.claude/todo-config.json`。
6. 告知用户："📌 TODO 文件已设置为 `<路径>`。之后说「修改 todo 文件」可随时更换。"

### 重新配置流程
当用户想更换 TODO 文件时：

1. 若已有配置，显示当前路径："当前指向 `<路径>`。"
2. 询问新路径，或确认是否新建文件。
3. 验证路径是否存在，不存在则提议创建。
4. 更新 `.claude/todo-config.json`。
5. 确认："✅ TODO 文件已更新为 `<新路径>`。"
6. 若之前已有不同路径，询问："是继续添加需求，还是只是更新配置？"

## 阶段二 — 澄清需求

> **方法论**：参照 `superpowers:brainstorming` 的访谈风格，但深度控制在 backlog 条目级别，不是完整设计。目标是写出一条清晰、可执行的需求条目，不是规格文档。够写了就停。

**先探索上下文。** 在提问之前，先读相关代码或文档，了解这块功能的现状。这样可以避免问自己能回答的问题，并帮助提出更精准的澄清问题。

**然后一次只问一个问题。** 不要把多个问题打包在一起。优先给出选择题——比开放题更容易回答。等用户回答后再问下一个。

按顺序覆盖以下方面，**已从用户描述或上下文中明确的可跳过**：

1. **核心需求** — 要做什么？解决什么问题？（如果用户已描述清楚，确认理解即可。）
2. **背景 / 动机** — 为什么现在需要做？有没有依赖、约束或触发事件？
3. **验收标准** — 怎么判断做完了？（1–3 条具体、可测试的条件，写成 checkbox）
4. **优先级** — 是否紧急？与其他需求有没有先后依赖？

**提问方式**（参照 brainstorming 风格）：
- 每条消息只问一个问题，无例外
- 优先给选择题："这个需求更偏向 (A) 交互体验优化、(B) 功能缺失补全，还是 (C) 性能问题？"
- 使用用户自己的语言和领域词汇
- 若需求范围过大，提示拆分："这个听起来可能跨好几个独立的需求，要不要先拆一下？"

经过 2–4 轮问答后，汇总理解，展示格式化的条目预览。在动文件之前，先问："这样写对吗？"

## 阶段三 — 冲突与重复检查

写入之前，加载 TODO 文件，扫描是否有与新需求重叠的已有条目：

- **重复**：同一件事已经记录过
- **冲突**：新需求与现有条目矛盾或互相抵消
- **覆盖**：现有条目已将此需求包含为子任务或验收标准

用关键词匹配加语义判断（不只是精确字符串匹配）。若发现疑似重叠，展示给用户：

> "发现一个可能重叠的已有条目：
> **[GUI-003] 右侧面板响应式宽高** — ……
> 这是同一件事、子任务，还是独立需求？"

由用户决定如何处理：跳过、替换、合并，或作为子任务挂载。

只有用户确认新条目是独立需求后，才继续进入阶段四。

## 阶段四 — 打标签与分组建议

### 类型标签（选一个，以主要意图为准）

| 标签 | 适用情况 |
|------|----------|
| `feature` | 新增用户可感知的功能（原来没有这个能力） |
| `bug` | 某个预期行为没有正常工作 |
| `refactor` | 改善代码质量/结构，对外行为不变 |
| `performance` | 速度、内存或资源消耗的改进 |
| `ux` | 已有功能的交互/视觉体验改进，功能本身不变 |
| `docs` | 仅文档变更 |
| `dx` | 工具链、构建、测试基础设施、开发体验 |
| `security` | 安全漏洞修复或安全加固 |

> 区分 `feature` vs `ux`：加一个新按钮是 `feature`；把现有按钮改得更好用是 `ux`。

### 模块标签（选一或两个）
从项目实际目录/组件结构推导，不要自造名称。检查：
- 顶层目录（如 `electron/`、`core/`、`services/`、`scripts/`、`cli`）
- `package.json` 中的包名
- TODO 文件中已有的模块标签（优先保持一致）

**优先使用反映功能领域的名称**（如 `gui`、`api`、`cli`），而非技术基础设施名（如 `electron`、`node`）——除非项目本身就用技术名作为模块标识（可从已有 TODO 条目或 CLAUDE.md 判断）。

若需求真正横跨两个模块，使用两个标签。若超过两个，选承载最主要变更的那个。

### 优先级标签（选一个）

| 标签 | 判断标准 |
|------|----------|
| `P0-紧急` | 阻塞当前开发流程、生产问题、或有明确截止日期 |
| `P1-高` | 对下一个里程碑有明显影响，用户可感知的重要缺失 |
| `P2-中` | 有价值但不紧迫，正常开发队列（**默认**） |
| `P3-低` | 锦上添花，可以很晚再做 |

除非用户明确表示紧急或重要，否则默认 `P2-中`。

### 分组建议

打完标签后，扫描已有 TODO 条目，按以下信号强弱判断是否可以批量开发：

1. **影响范围重叠**（最强信号）— 两个需求都会修改相同文件或系统层（如都改动 `electron/src/renderer/`）。合并做可减少上下文切换和合并冲突。
2. **同模块 + 同类型** — 如同属 `gui` 模块的多个 `ux` 改进，可以复用相同的测试环境和视觉上下文。
3. **依赖关系** — 一个是另一个的前置条件。标注为顺序依赖，而不是"一起开发"。

**不建议分组的情况：**
- 优先级差异明显（不要让 P1 等 P3 凑批）
- 影响范围完全不同（如 `core/` 改动 vs `docs/` 改动）
- 其中任一需求本身还不清晰

若有分组候选，给出提示，但仅供参考，不阻塞写入：

> "💡 **[GUI-001]** 和这条需求都涉及 `electron/src/renderer/`，可以考虑放在同一个开发会话里做。"

## 阶段五 — 写入条目

将新条目追加到 TODO 文件的 `## 🚧 待开发` 区块末尾，使用以下格式：

```markdown
### [MODULE-NNN] 标题（简短、祈使句式，≤30 个汉字或 60 个英文字符）
**标签**: `类型` `模块` `优先级`  
**影响范围**: 一行描述涉及哪些文件或系统层  
**创建日期**: YYYY-MM-DD

简要说明要做什么、为什么做（2–4 句）。

**验收标准**:
- [ ] 验收条件 1
- [ ] 验收条件 2
- [ ] 验收条件 3（可选）

---
```

**ID 生成规则：**
- 前缀：模块标签的大写形式（如 `gui` → `GUI`、`core` → `CORE`、`cli` → `CLI`）
- 序号：扫描同前缀的已有 ID，取最大值 + 1，从 `001` 起
- 若需求横跨多个模块，用主要模块的前缀

写入后确认："✅ 已将 **[MODULE-NNN] 标题** 写入 `<文件路径>`。"

---

## 核心原则

- **未经用户确认格式化预览，绝不写入文件。**
- **阶段二不可跳过** — 哪怕只问一个问题，也能避免写入模糊无法执行的条目。
- **条目要可执行** — 每条需求应清晰到开发者拿到就能开始动手，不需要再追问。
- **不要写成设计文档** — 这是 backlog，详细规格属于 `docs/`。每条条目控制在 200 字以内。
- **使用用户的语言** — 用户用中文写，就用中文回应和写入。用英文写，就用英文。
