# 内置方法论：角色分类法

适用于知识型项目（研究、写作、竞赛准备、案例分析）。按文档在工作流中的**角色**而非类型组织。

---

## 核心原则

同一形式的文档可能属于不同角色：一份"会议记录"可能是工作过程，也可能是最终演讲稿。按类型分类会掩盖这种差异；按角色分类则清晰。

---

## 分类

### source (`source/`)

定义：读者需要追溯原始材料、核实数据来源时来这里。

包含：外部获取的原始材料（数据集、案例文章、PDF报告、理论工具书）。

排除：你自己写的分析和结论，哪怕是基于这些材料写的。

---

### analysis (`analysis/`)

定义：读者需要查阅项目的核心判断、框架、结论时来这里。

包含：你主动维护的思考结论（分析框架、诊断报告、战略文档、理论映射）。

排除：原始材料、过程草稿、面向外部的交付物。

---

### output (`output/`)

定义：读者是项目之外的人（评委、读者、客户），需要接收交付物时来这里。

包含：面向外部受众的交付物（演讲稿、PPT、发布文章、报告）。

排除：内部分析文档、原始材料、过程记录。

---

### working (`working/`)

定义：读者需要回溯决策过程、查阅历史迭代时来这里。

包含：产生结论的中间过程（会议记录、早期草稿、被替换的版本、存档内容）。

排除：当前有效的分析结论（那些放 `analysis/`）。

---

## 分类决策流程

```
这是外部获取的原始材料（只读，不修改）？
  是 → source/

这是写给项目之外的人看的交付物？
  是 → output/

这是你的思考结论，当前仍在维护？
  是 → analysis/

这是产生结论的中间过程，或已归档的旧版本？
  是 → working/
```

---

## 命名规则

| 场景 | 规则 | 示例 |
|------|------|------|
| 分析文档 | `kebab-case.md`，按主题命名 | `financial-mechanism.md` |
| 会议记录 | `YYYYMMDD-topic.md` | `20260531-strategy-sync.md` |
| 归档版本 | `YYYYMMDD-v1-topic.md` | `20260401-v1-proposal.md` |

---

## 二级组织

role 确定后，内部可按主题或受众进一步细分：

```
output/
  presentation/    # 按受众（评委）
  article/         # 按受众（公众号读者）

source/
  financials/      # 按主题（财务数据）
  theory/          # 按主题（理论工具箱）
```

二级目录按主题或受众命名，不再按类型命名。

---

## 验证

> 新人打开目录，能否在 10 秒内找到：
> - 核心结论 → `analysis/`
> - 原始材料 → `source/`
> - 最终交付物 → `output/`

能 → 结构合理。不能 → 角色没分清。
