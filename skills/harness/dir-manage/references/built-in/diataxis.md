# 内置方法论：Diátaxis

适用于软件项目的技术文档目录（通常是 `docs/`）。

---

## 分类

### tutorials (`tutorials/`)

定义：读者想从零开始学习，愿意跟着做一遍以理解原理。

包含：引导式步骤，有明确的学习成果，边做边解释。

排除：孤立的原理解释、完整的参数参考。

---

### how-to (`how-to/`)

定义：读者有一个具体任务要完成，知道自己想做什么，需要最短路径。

包含：完成任务的操作步骤，结果导向。

排除：为什么这样做的解释，背景原理。

---

### reference (`reference/`)

定义：读者需要查找某个事实、参数或规范，带着具体问题来查。

包含：稳定的事实、API 定义、配置参数、格式规范。

排除：操作步骤、设计原理、历史背景。

---

### explanation (`explanation/`)

定义：读者想理解为什么，想建立心智模型，带着好奇心而非具体任务来。

包含：背景、原理、设计原因、权衡取舍。

排除：操作步骤、完整参考信息。

---

## 分类决策流程

```
内容包含从零开始的引导式练习？
  是 → tutorials/
  否 → 包含完成某个具体任务的操作步骤？
          是 → how-to/
          否 → 包含稳定的事实、规范或 API 定义？
                  是 → reference/
                  否 → explanation/
```

## 命名规则

| 场景 | 规则 | 示例 |
|------|------|------|
| 长期文档（四类主目录） | `kebab-case.md`，不带日期 | `deploy.md`、`api-auth.md` |
| 有时间意义（adr/rfcs 等） | `YYYY-MM-DD-topic.md` | `2026-04-23-dag-rewrite.md` |
| 根目录索引 | `SCREAMING_SNAKE.md` | `INDEX.md` |

---

## 扩展分类

四个分类能覆盖绝大多数文档需求。只有当内容**确实无法归入任何一类**时，才考虑增加新目录。

可接受的扩展示例：

| 目录名 | 适用场景 | 与四类的区别 |
|--------|---------|-------------|
| `adr/` | 已落地的架构决策记录 | 不可修改、有时间戳、记录决策过程 |
| `rfcs/` | 提案、尚未实现的设计 | 状态是「待决策」 |
| `archive/` | 历史文档，只读归档 | 不再维护 |

新目录必须满足：
- 不与四类重叠
- 至少 2～3 篇同类内容才值得单独建目录
- 在 `INDEX.md` 中建立独立分区
