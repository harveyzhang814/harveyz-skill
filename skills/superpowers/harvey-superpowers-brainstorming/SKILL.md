# Brainstorming - 设计前探索

## 概述

帮助将想法转化为完整的设计文档和规格说明。在开始实现前，必须先理解需求并获得用户批准设计。

## 核心原则：HARD-GATE

```
在获得用户对设计的批准之前，禁止：
- 调用任何实现技能
- 编写任何代码
- 搭建项目脚手架
- 采取任何实现行动
```

**每个项目都必须经过此流程。** 即使是看似简单的任务，也必须先设计再实现。"简单"项目往往因为未经验证的假设而浪费最多时间。

## 流程概览

```
1. 探索项目上下文 → 检查文件、文档、最近提交
2. 提出澄清问题（一次一个）
3. 提出 2-3 个方案（含权衡和建议）
4. 展示分节设计 → 获得用户对每个部分的批准
5. 编写设计文档 → 保存到 docs/superpowers/specs/
6. 自检规格 → 检查占位符、矛盾、歧义、范围
7. 用户审查 → 等待用户批准书面规格
8. 转入 writing-plans → 创建实施计划
```

## 详细流程

### 1. 探索项目上下文

先了解当前项目状态：
- 检查相关文件（read 工具）
- 查看现有文档
- 查看最近的 git 提交历史

### 2. 澄清问题

一次问一个问题，理解：
- **目的** — 用户想解决什么问题？
- **约束** — 有哪些限制（技术、时间、资源）？
- **成功标准** — 如何衡量成功？

### 3. 提出方案

提出 2-3 个不同方案，包含：
- 每个方案的权衡
- 你的建议和理由
- 用对话方式呈现选项

### 4. 分节展示设计

根据复杂度调整每个部分的长度：
- 简单项目：几句话
- 复杂项目：每个部分 200-300 字

覆盖内容：架构、组件、数据流、错误处理、测试策略

### 5. 编写设计文档

保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

规格自检（内联修复）：
1. **占位符扫描**：是否有"TBD"、"TODO"、不完整部分？
2. **内部一致性**：各部分是否矛盾？
3. **范围检查**：是否聚焦于单一实施计划？
4. **歧义检查**：是否有多种解释的需求？

### 6. 用户审查

> "规格已保存到 `<path>`。请在开始编写实施计划之前审查并告诉我是否需要修改。"

等待用户响应。如有修改，进行修改后重新自检。

### 7. 转入实施

调用 `writing-plans` 技能创建详细实施计划。

## 关键原则

- **一次一问** — 不要用多个问题压垮用户
- **多选优于开放** — 尽量提供选项
- **YAGNI 原则** — 从所有设计中剔除不必要功能
- **探索替代方案** — 选定前提出 2-3 个方案
- **增量验证** — 展示设计，获得批准后再继续
- **灵活调整** — 有疑问时返回澄清

## 大型项目分解

如果项目涵盖多个独立子系统：
1. 立即标记项目需要分解
2. 帮助用户将项目拆分为子项目
3. 确定独立部分及其关系
4. 确定构建顺序
5. 对第一个子项目进行正常设计流程
6. 每个子项目独立获得 spec → plan → implementation 循环

## 设计文档示例结构

```markdown
# [功能名称] 设计文档

## 概述
[1-2 句话描述]

## 背景
[为什么需要这个功能]

## 用户故事
[功能如何被使用]

## 架构设计
[主要组件和关系]

## 数据流
[数据如何流转]

## 错误处理
[异常情况如何处理]

## 测试策略
[如何验证功能]

## 风险和缓解
[已知风险及应对]
```

## 触发条件

当用户提出以下请求时触发此技能：
- "实现一个..."
- "添加...功能"
- "帮我做..."
- "我想让系统..."
- 任何需要设计和批准的新需求

**禁忌：** 不要跳过设计阶段直接实现。即使是"快速修复"或"简单改动"也需要先理解意图。
