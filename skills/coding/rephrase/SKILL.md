# rephrase — 单轮改写澄清

对用户的一句话表述做**单轮**改写，输出更精确、更少歧义、更可执行的新版本。不预设"什么样的表述算合格"的标准或模板——由 Claude 依据具体语境自行判断改写是否到位，视情况而定，不是打勾清单。

---

## 触发条件

**主动调用：** 用户输入 `/rephrase` 或 `/rephrase <表述>`。

**不触发：**
- 不自动检测、不主动建议——必须用户手动调用
- 模糊或复杂任务的多轮追问式澄清 → 使用 `question-me`
- 需要生成结构化 Goal Prompt → 使用 `init-goal`

**不在范围内（边界）：**
- 不做多轮追问式澄清，只做一次改写
- 不强制补充验收标准、范围边界（除非改写过程中自然需要）
- 不做自动触发或自动建议
- 不保存改写历史或跨会话记录

---

## 执行步骤

### Step 1 — 确定输入来源

按优先级取待改写的表述：

1. `/rephrase <表述>` 带参数 → 直接使用该参数
2. 无参数 → 使用用户上一条消息作为待改写内容
3. 无参数且没有上一条用户消息 → 提示用户直接提供表述，不继续往下走

### Step 2 — 生成改写

基于上下文（对话历史、涉及的文件/代码，若有），生成一个更精确、更少歧义、更可执行的新表述：

- 消歧义（指代不明的"这个"、"它"、"那样"等）
- 补全隐含主语/宾语
- 明确动作对象（改什么文件、哪个功能、哪个范围）

这里刻意不给一张固定清单去照着填——像"每次都要消解代词""每次都要点名目标文件"这类规则，套在不需要的表述上就是无谓的形式主义，套不上真正需要的表述上又会漏判。改写该做到什么程度，取决于原表述本身缺了什么：如果它已经足够清楚，改写可能和原文几乎一样，这是正常结果，不必为了"看起来改过"而画蛇添足；也不要借改写之机替用户做他没要求的范围/验收标准决定。

只做一轮，不追问用户来回打磨。

### Step 3 — 自行判断可靠性

Claude 自行判断这个新表述是否"可靠"：是准确反映了用户意图，还是掺了没有依据的猜测。无预设标准，凭对当前语境的判断，不走固定检查表。

关键不在于"改写里是否存在任何未被原文明说的假设"——几乎所有改写都会补全一些没被逐字说出的内容，这不代表不可靠。真正要权衡的是**假设猜错的后果有多大、执行风险是否可控**：如果这个假设涉及在多个同权重候选里武断选一个（哪个文件、哪个服务、哪种截然不同的语义读法），一旦选错会导致做错事或改错范围，就判不可靠；如果只是无关紧要的实现细节，就算是"没明说"，只要执行风险低、改错了也容易回退，仍然可以判为可靠，直接往下走——单轮改写的意义就在于此，不是每次沾一点不确定性就退回去问。

### Step 4 — 按判断结果分支

**判断为不可靠：**
- 展示新表述
- 说明存疑之处（哪里可能理解错了、哪里是猜测）
- 让用户自行确认——是要修改还是直接按这个执行
- 等用户回应后再继续

**判断为可靠：**
- 展示新表述（保持透明，不隐藏改写过程）
- 不等待用户确认，直接把新表述当作用户输入去执行任务

---

## 不在范围内

- 多轮追问式澄清（属于 `question-me` 的职责）
- 强制补充验收标准/范围边界模板
- 自动检测触发或主动建议使用本 skill
- 预设"合格表述"的固定标准或清单
