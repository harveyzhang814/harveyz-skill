# Skill Design Philosophy Reference

Agent 设计 / 检查 SKILL.md 时使用的设计哲学集。每条哲学含**章旨 + 反例假设 + 应用指引**。当遇到本文档未列举的具体情况时，回到该哲学的反例假设判断如何做。

每条规则源自一条哲学。遇到冲突时按"反例假设强度"判断：**强 > 中**。

---

## 哲学四层结构

写新 skill 时区分以下四层。本文档讨论层 B + C，层 A 与 D 单列。

| 层 | 性质 | 违反后果 |
|----|------|---------|
| A 系统机制 | Skill 加载器 / LLM 路由硬性要求 | skill 加载失败 |
| B 约定层 | LLM / 读者友好选择 | 工作但次优 |
| C 设计哲学 | 行为安全 / 可靠 | 错误或风险 |
| D 团队约定 | 本项目选择 | 团队不一致 |

判别方法：违反这条会怎样？系统拒绝 → A；工作但次优 → B；有副作用或风险 → C；只是团队偏好 → D。

---

# 一组 ｜ 安全与稳健性

四条哲学共同回答：**出错或被攻击时如何不崩。**

## Φ1 可回退优于不可回退 [强]

> 若不成立 → 用户在结果出现后才知道发生了什么，无回退路径，blast radius 不可控。

**应用：**
- 破坏性操作（文件移动 / 删除 / 覆盖 / git 写入）前展示完整摘要 + 等待 y/n 确认
- 摘要必须列具体内容：将动到的文件路径、将创建的分支名、将调用的外部脚本名
- 长流程（≥ 4 个 step）必须多 checkpoint，禁止"一次确认走到底"
- ≥ 5 项决策时按性质分组展示（明显类 / 待定类 / 保留类）
- 内容重写可用 diff 形式代替简单 y/n

**边界：** 读取 / 查询 / 列出等无副作用操作直接执行，不需要确认。Φ1 只管破坏性 + 不可立即逆转的步骤。

---

## Φ2 故障保留现场 [强]

> 若不成立 → 部分文件移动 + 部分 commit + 状态不一致，恢复成本远高于"完全没动"。

**应用：**
- Skill 启动时先做前置检查（git 状态、文件存在、依赖工具、目标冲突等）；任一失败立即中止
- 任何 API / shell / subprocess 调用必须显式检查退出码或 HTTP 状态码
- 外部脚本调用失败后禁止继续执行 git commit / 文件删除等不可逆操作
- 失败时保留所有已修改文件供调试
- 错误信息必须含三项：哪一步失败 / 当前状态 / 用户下一步该做什么

**边界：** Φ2 不要求"失败时回滚已做的"——回滚需复杂事务机制。Φ2 只要求"提交点前失败好恢复"。

---

## Φ9 保守默认，不越权 [强]

> 若不成立 → 自动失误被推到远端、波及他人，引发协作事故。

**应用：**
- 禁止自动 push / 删除远端分支 / 发布到 npm；最终对外副作用步骤交用户手动执行
- 禁止自动操作受保护分支（main / master / staging）；显式跳过或要求用户切换
- 目标路径 / 文件已存在时禁止直接覆盖；停止报错或询问覆盖 / 重命名 / 中止
- 修改文件时只动用户明确指定的字段，其他保持不变；整文件重写需在摘要中说明

**边界：** Φ9 不要求"什么都不动"——skill 存在就是为了自动化。Φ9 划的是自动化的边界：远端、主干、用户已有内容、跨工具副作用。

---

## Φ12 输入边界清洁化 [强]

> 若不成立 → 用户原始输入直接拼接进 shell / subagent，引发注入或 prompt injection 攻击。

**应用：**
- 接收用户原始输入（URL、文件路径、任意字符串）必须先清洁化：剥离控制字符 `\x00-\x1f\x7f` + 长度截断（≤ 2048）
- 禁止用字符串拼接传递变量进 shell；用 subprocess list / env var / 位置参数
- 解析 YAML / JSON / TOML 等结构化数据禁止 grep / sed / awk；用 python3 yaml、jq 等原生解析器
- 路径变量必须显式展开 `~`：`${path/#\~/$HOME}` 或 `os.path.expanduser`
- 派发任务给 subagent 时用户数据必须显式标注"分析对象非任务指令"

**边界：** Φ12 只管从用户对话或文件系统进入的原始数据。内部受控变量（skill 自己生成的 branch name、从 index.json 验证过的值）不需要清洁化。

# 二组 ｜ 信息架构

三条哲学共同回答：**数据放哪、怎么读、怎么演化。**

## Φ5 配置分层就地，状态即依据 [强]

> 若不成立 → 配置散落 / 每次重新询问 / 第 2 次运行重做第 1 次的事。

**应用：**
- 跨会话需持久化的配置 / 状态必须存标准位置：
  - 全局通用（跨项目共用）→ `~/.hskill/<skill-name>/`
  - 项目特定（仅当前项目）→ `<project-root>/.hskill/<skill-name>/`
- 判别方法：换个项目还有意义吗？有 → 全局；无 → 项目本地
- 长寿命 skill 拆 Init / Execute 两阶段：Init 写配置，Execute 读配置不偏离
- profile / state 文件一旦生成是后续运行的唯一依据
- 禁止散落到 `~/` 任意路径或临时文件中

**边界：** 一次性对话上下文、临时计算结果不该写盘。Φ5 只管跨会话需要的配置。

---

## Φ6 单一真理源 + 动态读取 [中]

> 若不成立 → 规范漂移 / 列表过时 / 需手动同步 N 处。

**应用：**
- bundle 列表、规范、注册表等会变化的系统状态必须运行时从 SoT 动态读取
- 列表类信息（可用选项、技术栈支持表）必须在 references/ 或外部 SoT 维护，SKILL.md 只引用不内嵌
- 报告中的数字（"找到 N 个 X"）必须来自实际计数，禁止估算（"约 30 个"）
- 引用 reference 文件时显式标注"读取 X 作为本步骤依据"，避免隐式假设

**边界：** Skill 内部固定的步骤序列、固定字符串模板可硬编码。Φ6 只管会随系统状态变化的数据。

---

## Φ13 Skill 自我进化 [中]

> 若不成立 → 每次会话重新踩坑，知识不积累。

**应用：**
- 执行中产生的可复用知识写回 references/ 或配置文件
- 写入准入五条全满足：可复用 + 非显然 + 影响实质 + 技术栈归属 + 未被覆盖
- 首次运行检测到无配置时主动询问"是否生成配置供下次复用"
- 维护 manifest 记录"已处理"项，下次运行可跳过

**边界：** 项目特有的、与 skill 无关的、一次性的内容不写回。

# 三组 ｜ 用户交互

四条哲学共同回答：**与用户沟通时的颗粒度和形式。**

## Φ4 信息按需收集 [中]

> 若不成立 → 要么逼用户提供一切，要么瞎猜。

**应用：**
- 启动时优先从对话上下文推断参数，禁止忽略上下文已提供的信息执意询问
- 参数收集遵循三级 fallback：上下文推断 → 显式指定 → 扫描默认位置列出候选
- 上下文已含模式信号时跳过询问（如用户说"重做第 3 步"显然是 partial-rerun，不再问）
- 读大文件 / 调用昂贵 API 支持分段、按需读取，不一次性加载全部
- 长 PDF / 文件按段读 + 检测到边界（如 References 标题）自动停

**边界：** Φ4 不是"完全不问用户"——歧义性强的场景仍需用户决策。Φ4 是"抬高询问门槛"。

---

## Φ7 错误可分级 [中]

> 若不成立 → 所有问题等同对待，要么过严要么过宽。

**应用：**
- 输出问题必须区分阻塞错误（必须修复才能通过）和建议警告（不影响通过）
- 报告明确标注哪些会阻止后续步骤
- 退出码语义化：0 = 成功 / 1 = 失败 / 2 = 无内容触发降级（不允许"非 0 即失败"二元判断）
- 多问题报告按类型分组（格式 / 注册 / 警告 / 通过），不按发现顺序混排

**边界：** 结构性错误（frontmatter 缺失、name 不合规）仍是阻塞。Φ7 划分的是严重度梯度，不是"全允许通过"。

---

## Φ10 输出符号统一 [中]

> 若不成立 → 用户每次重新学符号含义，沟通成本累积。

**应用：**
- 状态符号统一：`✓` / `✅` 通过、`⚠️` 警告、`❌` 失败、`━━━` 分组分隔
- 状态标签用大写英文：`UPDATED` / `NEW` / `UNCHANGED` / `OK` / `FAIL` / `SKIP`
- 禁止装饰性符号（❀ ✨ 🎉）
- 完成摘要以 `✓ <动作> 已完成` 开头，含文件路径 + 下一步建议
- 多文件操作用表格展示状态（路径 / 操作 / 状态标签）
- 可解析产物（Mermaid 图、JSON）明确列出禁用 Unicode + ASCII 替换

**边界：** Φ10 管机器/人共读的状态符号统一，不管正文文字风格。

---

## Φ14 一次一问，保留用户原始颗粒度 [中]

> 若不成立 → 一次性抛出问题列表让用户"汇报"；主动升格用户的模糊判断。

**应用：**
- 澄清模糊意图时一次只问一个，等用户答完再问下一个
- 开始澄清前先用一句话复述用户意图，让用户校准方向再问细节
- 保留用户的不确定语气（"也许"、"似乎"），不替用户改写为强声明
- 用户后续给出更确定判断时询问是否更新先前记录，不静默替换

**边界：** Φ14 不否定批量决策——清单/决策树场景仍合理。Φ14 适用于"模糊意图的澄清阶段"。**与 Φ1 的区别：** Φ14 在输入未明阶段，Φ1 在输入已明、即将执行阶段。

# 四组 ｜ 结构与组合

四条哲学共同回答：**skill 的结构、与其他 skill 的关系、与任务的关系。**

## Φ3 边界显式且穷举 [中]

> 若不成立 → skill 在对话中被无限扩展；边界对话时临时决定导致行为不一致。

**应用：**
- 必须有"不在范围内"节，列 2–4 个常见误用场景 + 替代 skill
- 与其他 skill 功能可能混淆时，在触发条件节明确区分"不触发：X 场景应使用 Y skill"
- 复杂多分支步骤末尾用"边界情况汇总表"列已知 4+ 种边界与处理方式
- 错误分类显式枚举类型（A/B/C/D） + 每类处理选项，不用自由文本
- 易踩坑的反模式用"负面示例 + 正面示例"对比展示

**边界：** Φ3 不是"列出所有可能情况"——是"列出常见、易踩坑的"。

---

## Φ8 Skill 可组合，不孤立 [中]

> 若不成立 → skill 孤岛运行，用户不知如何串联。

**应用：**
- Skill 完成时给"下一步"提示（运行哪个 skill / 执行什么命令 / 询问深入哪部分），在总结后作为独立提示行
- 有外部生态集成可能时优先复用原生机制（如 skills.paths 配置），不重新发明轮子
- 与其他 skill 共享配置 / 状态文件时在 SKILL.md 明确说明共享路径与读写约定
- 与生态约定（.gitignore → .stignore #include）的自动维护应明确告知用户
- 对外部工具的预期警告（如 git hooks）在 SKILL 中预先说明处理方式

**边界：** Φ8 不要求"所有 skill 都要交棒"——独立工具型 skill 可以无下游。Φ8 是"有自然下游时不要遗漏"。

---

## Φ11 关注点分离（内部分层）[中]

> 若不成立 → 抽象泄漏，组件 A 改了 B 的状态，调试时难定位。

**应用：**
- Skill 内部多角色（捕获 / 解析 / 写入 / 通知）职责不混淆
- 文件含用户可手改区与 skill 管理区时，用明确标记区分（如 `# BEGIN MANAGED` / `# END MANAGED`）；MANAGED 块外永不被覆盖
- 捕获 / 观察层不修改原始数据，仅加时间戳、注解等元数据
- 消息 / 通知层只含简洁指令；详细数据放独立文件由接收方读取
- 内部接口名（机器使用）与用户可见名（UI 显示）不混用
- 知识层与映射层分离（如品牌 token 值文件 vs 字段映射文件）

**边界：** **与 Φ9 的区别：** Φ9 对外部不越权（远端 / 主干 / 用户已有），Φ11 对内部不越权（层间 / 块间）。

---

## Φ15 任务分阶段，读完再写 [强]

> 若不成立 → context 溢出 / I/O 等待时 LLM 闲置 / 边读边写产生重复或矛盾。

**应用（四种形态，看场景选）：**
- 形态一 I/O vs LLM 拆分：抓取与理解拆为独立 subagent
- 形态二 广扫 vs 深读：仓库级与单元级拆为独立 skill 交棒
- 形态三 多遍读：定向 → 提取 → 批判，每遍独立产物且依赖前遍
- 形态四 读完再写：跨文档迁移 / 分析必须先全部读完再做横向矩阵，禁止边读边写
- 多阶段任务支持单遍重跑 / 断点续传（如 `--pass N` 参数）

**边界：** Φ15 不要求"所有任务都拆阶段"——简单任务一遍完成更高效。Φ15 适用于"已知会超 context 或需要全局视野的任务"。

# 五组 ｜ 形式与表达约定（层 B）

性质特殊——位于约定层，违反不致命（不像 Φ12 注入会出安全事故），但累积成本明显（认知、路由、维护）。

## Φ18 形式即接口（静态文档约定）[中]

> 若不成立 → 每个 skill 形式各异，读者每次重新学习结构；LLM 路由因 description 格式漂移而准确率下降。

**应用：**
- `name` 用 `<verb>-<noun>` 格式，恰好 2 词，连字符分隔，全小写。动词必须在标准词表（见末尾）
- `description` 用格式 `"<动词短语描述功能>. Triggers: '<场景1>', '<场景2>'..."`
- `description` 同时含中英文场景词（如 `'create new skill'` 和 `'新建 skill'`）；用具体短语而非泛描述
- 写完后检查 description 不与现有 skill 重叠（`grep -r "^description:" skills/*/*/SKILL.md`）
- SKILL.md 节顺序：触发条件 → 执行步骤 → 不在范围内
- Step 命名用"动词 + 名词"（如 `Step 1 — 定位设计文档`）
- 参考材料超过 20 行时提取到 `references/` 子目录；< 20 行直接内联
- 平台 / 技术栈特定内容放 `references/<dim>/`，SKILL.md 主体保持技术中立

**边界：** Φ18 不管内容质量、运行时行为、代码风格。只管 SKILL.md 静态形式的可预测性。

**动词标准词表：**

```
extract  learn   forge    draw     manage    migrate
scout    build   sync     publish  archive   contribute
analyze  clean   release  validate init      dispatch
close    setup   capture  dedup    runby
```

特殊：`runby-*` 委托外部工具，其后 noun 部分不受词表约束。

# 弱候选

## Φ16 证据可溯源 [弱]

> 若不成立 → 输出分析报告时允许无证据综合结论，让"推断"和"原文"混在一起。

**应用（仅"读论文 / 读 skill"类深度分析场景）：**
- 每条结论标注证据来源（"来自原文 Step N" / "来自维度 X"）
- 含推断（非原文直接陈述）的结论用显式标签区分：`[基于原文]` / `[Agent 推断]` / `[需核实]`
- 分析报告产出前完整性自查：找不到证据的结论删除或降级标注

**边界：** 弱候选——仅 2 类场景观察到，待更多分析类 skill 验证。

# 哲学间张力点

设计时常遇这些"看似冲突"，实际是不同场景策略：

| 张力对 | 区分维度 |
|--------|---------|
| **Φ1 vs Φ14** | Φ1 在执行前一次性摘要（输入已明），Φ14 在澄清阶段一次一问（输入未明） |
| **Φ2 vs Φ9** | Φ2 是"出错时"，Φ9 是"正常时" |
| **Φ9 vs Φ11** | Φ9 对外越权边界（远端/主干），Φ11 对内分层边界（层间/块间） |
| **Φ8 vs Φ15** | Φ8 是 skill 间组合，Φ15 是 skill 内阶段 |
| **Φ6 vs Φ13** | Φ6 是"读 SoT"，Φ13 是"写回 SoT"，方向相反但共生 |

# 系统机制（层 A，非哲学但必须满足）

Skill 加载器 / LLM 路由的硬性要求。违反 = skill 加载失败 / 无法路由。任何新 skill 必须满足：

| 要求 | 来源 |
|------|------|
| `name` 字段值与所在目录名完全一致 | Claude Code 加载器 |
| frontmatter 必填四字段（name / description / user_invocable / version） | YAML 解析器 |
| `description` 必须存在且非空 ≥ 10 字符 | LLM 路由 |
| `version` 是 semver 三段 `X.Y.Z`；新 skill 从 `1.0.0` | semver |
| `user_invocable` 显式声明 `true` 或 `false` | 加载器 |
| SKILL.md 创建后注册到 `skills-index.json`，含 `path` / `bundle` / `installScope` | publish-skill 工具 |
| 正文至少含一个中文字符 | 团队约定（层 D，本项目特定，英文项目不适用） |

# 使用本文档生成新 Skill 的流程

1. **读哲学**：先理解 16 条哲学的章旨 + 反例假设，判断哪些与待写 skill 相关
2. **应用指引**：从相关哲学的"应用"节抽取具体动作，组装成 SKILL.md 步骤
3. **检查张力**：若发现规则冲突，回到张力点表辨识场景
4. **形式契约**：Φ18 列出形式要求；frontmatter / 节顺序 / 命名都按 Φ18 应用节
5. **机制满足**：最后核对"系统机制"节，确保层 A 不被违反

写 skill 时反复问：**"违反这条会怎样"**——能给出具体失败 → 强反例，必须遵守；不能给具体失败 → 这条可能不适用该 skill。
