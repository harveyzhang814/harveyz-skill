### B. Guide 有 / 哲学衍生中无（哲学体系遗漏，13 条）

这些规则是 guide 有但哲学体系没覆盖：

| Guide | 类型 | 哲学体系是否应补 |
|-------|------|-----------------|
| G1 命名 `<verb>-<noun>` | **元规则** | 不必——属于命名约定，超出"设计行为"范畴 |
| G2 frontmatter 字段 | **元规则** | 不必——属于格式约定 |
| G3 description 英文 ≥ 10 字符 | **元规则** | 不必——属于格式约定 |
| G4 正文含中文 | **元规则** | 不必——团队约定 |
| G5 description 格式模板 | 表达规范 | **应补**——可纳入新哲学"形式即设计" |
| G6 检查 description 重叠 | 操作规范 | **应补**——可纳入 Φ3 边界穷举 |
| G7 节顺序 | 表达规范 | **应补**——结构化让读者一眼看出执行路径 |
| G9 Step 名动词 + 名词 | 表达规范 | **应补**——同上 |
| G12 references/ 超过 20 行提取 | 文件组织 | **应补**——可纳入 Φ11 关注点分离（SKILL.md vs 参考资料） |
| G13 平台特定内容放 references/ | 文件组织 | **应补**——同上 |
| G16 注册到 skills-index.json | **元规则** | 不必——属于 publish-skill 职责，独立于设计哲学 |

**Gap 性质分析：**

13 条遗漏分两类：

#### B1. 元规则类（5 条：G1/G2/G3/G4/G16）

属于 **skill 系统的元规范**——告诉你 skill 长什么样（命名、frontmatter、注册），不告诉你 skill 做什么。

**结论：这是哲学体系的合理边界。** 哲学回答"如何设计行为"，元规则回答"如何打包成 skill"。两者是不同抽象层。

**建议：** 元规则继续放 guide 独立节，不强行纳入哲学体系。可在文档说明"以下是元规则，不由哲学衍生"。

#### B2. 表达 / 文件组织类（6 条：G5/G6/G7/G9/G12/G13）

属于 **形式规范**——影响"如何写"而非"如何做"。

**结论：这是哲学体系的真实遗漏。** 我的 15 条哲学偏重行为设计，对"文档形式"重视不足。

**建议：** 考虑新增哲学 **Φ18 形式即设计**（候选）：

> Skill 是"文档先于实现"的产物——SKILL.md 本身就是设计；表达规范（节顺序、命名风格、文件组织）不是装饰，是设计的一部分。读者通过 SKILL.md 的形式就能推断执行路径。

下属规则：
- G5 description 格式模板
- G6 检查 description 重叠
- G7 节顺序
- G9 Step 命名
- G12 references/ 提取阈值
- G13 平台特定文件组织

**反例假设：** 若不成立 → SKILL.md 形式各异，读者每次重新学习结构。

**反例假设强度：** 中

**覆盖：** 几乎所有 30 个 skill（虽不显式声明但实际遵守）。

## 四、哲学完备性结论

### 4.1 双向 Gap 性质对比

| 方向 | 数量 | 性质 |
|------|------|------|
| Guide → 哲学 缺 | 13 条 | 5 条元规则（合理边界）+ 6 条表达规范（应补哲学 Φ18） |
| 哲学 → Guide 缺 | 20 条 | guide 实际遗漏，应补充 |

### 4.2 哲学体系评分

**完备性：** 75%（覆盖 guide 现有规范的 16/28 ≈ 57%；剩余 12 条中 5 条是合理元规则，7 条是潜在 Φ18）

**前瞻性：** 哲学衍生出 20 条 guide 没有但合理的规则，说明哲学体系超越现状。

**主要遗漏：** 表达 / 文件组织规范（应补 Φ18 形式即设计）

### 4.3 Guide 评分

**核心覆盖：** 中等——核心安全（Φ12 拼接禁忌）、配置（Φ5）、输出（Φ10）都有，但 Φ2 故障保留、Φ7 错误分级、Φ11 关注点分离、Φ14 一次一问、Φ15 任务分阶段几乎空白。

**等级偏弱：** 6 处 SHOULD 应升为 MUST（如配置存 .hskill、视觉符号统一）。

**严重缺失：** prompt injection 防御（R12.3）+ 自动 push 禁止（R9.1）+ MANAGED 块（R11.2）三条 MUST 完全空白。
