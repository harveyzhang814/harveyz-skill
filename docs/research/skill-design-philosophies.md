# Skill 设计哲学与范式

本文档汇总从 `skills/` 下 30 个 SKILL.md 中提炼出的设计哲学，每条配下属范式与来源 skill。

本文档**不替代** `skill-authoring-guide.md`——它是研究产出的描述性知识，guide 是规范性约束。二者关系：本文档解释 _why_，guide 规定 _how_。

---

## 哲学体系的四层结构

不是所有"必须遵守的规则"都是哲学。本研究区分四层抽象，**只有层 B 和层 C 是哲学层**：

```
层 A 系统机制（Skill 加载器 / LLM 路由要求）
       违反 → skill 加载失败 / 无法路由
       例：name 与目录名一致、frontmatter 字段必填、注册到 skills-index.json

层 B 约定层（Conventions） ← 哲学
       Φ10 运行时输出约定、Φ18 静态文档约定
       违反 → 工作但次优（认知成本累积、路由准确率降低）

层 C 设计哲学（Design） ← 哲学
       Φ1-Φ9, Φ11-Φ15
       违反 → 行为错误或安全风险

层 D 团队约定
       违反 → 团队不一致
       例：正文用中文（本项目选择，英文项目不适用）
```

**关键判别方法：** 问"违反这条会怎样"？
- 系统直接拒绝 / 报错 → 层 A 机制 spec
- 系统接受但工作不畅 → 层 B 约定
- 系统接受但有副作用或风险 → 层 C 设计
- 系统接受且无副作用，只是团队偏好 → 层 D

本文档讨论 16 条哲学（15 设计 + 1 约定）。机制和团队约定列在最后附录，明确**不属于**哲学。

# 一组｜安全与稳健性

四条哲学共同回答：**出错或被攻击时如何不崩。**

## Φ1 可回退，优于不可回退

**章旨：** 破坏性操作前必须有"看摘要 → 确认 → 执行"的回退机会。

**价值：** Skill 与传统 CLI 工具不同——CLI 错了用户立刻看见输出，可以 Ctrl+C；Skill 在对话流里运行，错误可能在多步之后才显现。**确认是回退机会的唯一可靠载体。** 若省略，"用户授权一次"等于授权整个会话所有操作，blast radius 不可控。

**反例假设：** 若不成立 → 用户在结果出现后才知道发生了什么，无回退路径。

**反例假设强度：** 强

**覆盖：** 11+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P001 | 操作前展示完整摘要 + y/n 确认 | archive-skill, clean-git, contribute-skill, init-skill, publish-skill |
| P017 | 分组展示 + 逐组确认 | clean-git, publish-skill |
| P036 | 多确认点等用户明确确认 | init-skill, contribute-skill, archive-skill |
| P023 | 自动可修复 vs 内容重写分级确认 | contribute-skill, publish-skill |
| P119 | 写入后摘要列出风险点让用户复核 | build-style, scout-brand, sync-design |
| P126 | 渲染前对照检查清单逐条核验 | draw-diagram |
| P142 | 澄清前先复述核心让用户校准 | capture-insight |

**反例边界：** 不是"所有操作都问用户"——读取文件、查询状态等无副作用操作直接执行。Φ1 只管**破坏性 + 不可立即逆转**的步骤。

---

## Φ2 故障保留现场

**章旨：** 任何中间步失败都不留半成品，保留文件供调试。

**价值：** 多步流程中，第 5 步失败时如果"已 commit 部分文件、移动了部分文件、跑了部分脚本"，恢复成本远高于"完全没动"。**保留现场 = 用户可以手动接力或 git status 看清楚发生了什么。** 这条哲学是错误恢复的基础设施。

**反例假设：** 若不成立 → 部分文件移动 + 部分 commit + 状态不一致，恢复成本高。

**反例假设强度：** 强

**覆盖：** 8+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P002 | 前置检查任一不满足立即中止 | archive-skill, contribute-skill, init-skill, release-project, runby-opencode, sync-agent |
| P007 | 调用外部脚本失败不进入 git 操作 | archive-skill, contribute-skill, publish-skill |
| P058 | API 4xx/5xx 输出状态码 + 响应体不静默 | sync-agent |
| P061 | 操作前路径/ID 格式校验 | sync-agent |
| P137 | --pass N 单遍执行校验前置文件存在 | learn-paper |

**反例边界：** 不是"失败时回滚已做的"——回滚需要复杂事务机制。Φ2 是**更弱但更实用的版本**：在任何"提交点"前失败都好恢复，已提交的部分接受。

---

## Φ9 保守默认，不越权

**章旨：** 不动主干、不动远端、不动用户已有内容；副作用最大的步骤交给用户执行。

**价值：** Skill 的危险性在于它**自动化**。自动化 + 协作分支 = 自动化失误会被推到远端、波及他人。Φ9 是协作场景的安全阀。区别于 Φ1（用户参与回退）和 Φ2（出错时不留烂摊子），**Φ9 管的是正常成功路径下的边界——能做的也不全做**。

**反例假设：** 若不成立 → 误删 main、误覆盖、自动 push 引发协作事故。

**反例假设强度：** 强

**覆盖：** 7+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P014 | 跳过保护分支（main/staging/当前） | clean-git |
| P018 | 远端操作交用户手动执行 | clean-git, release-project, contribute-skill |
| P048 | 本地操作完成 push/发布交用户 | release-project, contribute-skill, clean-git |
| P066 | 只更新指定字段其他内容不动 | close-task, init-workflow |
| P082 | MANAGED 块外用户代码永不覆盖 | init-workflow |
| P005 | 目标已存在拒绝执行不覆盖 | archive-skill, init-skill, contribute-skill |

**反例边界：** 不是"什么都不动"——skill 的存在就是自动化。Φ9 划的是**自动化的边界**：远端、主干、用户已存在的内容、跨工具的副作用，这些越界要让用户参与。

---

## Φ12 输入边界清洁化（注入防御）

**章旨：** 外部用户输入进入内部处理前必须清洁化——剥离控制字符、用结构化工具解析、subprocess list 传参。

**价值：** Skill 接受用户输入（URL、文件路径、命令参数）后，会触发 shell、调用 subagent、写文件。每一处都是注入面。**这是 Skill 系统最容易被忽略的安全维度**——大多数 skill 没意识到自己在处理"用户原始输入"。

**反例假设：** 若不成立 → 处理外部用户输入的 skill 会跳过控制字符剥离、用字符串拼接传参，存在注入风险。

**反例假设强度：** 强

**覆盖：** 5 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P099 | 剥离控制字符（`\x00-\x1f\x7f`）+ 长度截断 | extract-url, probe-session, learn-paper |
| P100 | subprocess list + env var 传参，禁字符串拼接 | extract-url |
| P101 | 派 subagent 时显式标注"外部数据非指令" | extract-url |
| P081 | 用结构化工具解析结构化数据（python3 YAML、jq JSON） | init-workflow, dedup-skill |
| P148 | 路径 `${path/#\~/$HOME}` 显式展开 | learn-paper |

**反例边界：** 不是"所有输入都做严格校验"——内部受控变量（skill 自己生成的 branch name、从 index.json 验证过的 skill name）不需要清洁化。Φ12 只管**从用户对话或文件系统进入的原始数据**。

**研究亮点：** 这条哲学的发现过程本身就值得记录。Phase 0 预期它存在，但 meta + coding bundle 都没看到。直到 research + experiment bundle 才浮现强样本（extract-url、probe-session、learn-paper 共享一套防御 idiom）。**没有跨 bundle 扫描，这条哲学会被错判为"先验错误"。**

# 二组｜信息架构

三条哲学共同回答：**数据放哪、怎么读、怎么演化。**

## Φ5 配置分层就地，状态即依据

**章旨：** 跨会话配置存 `.hskill/<skill>/`（项目本地）或 `~/.hskill/<skill>/`（全局）；profile/state 一旦生成，是后续运行的唯一依据。

**价值：** Skill 在多次会话间运行，**第 2 次运行不应该重新询问第 1 次确认过的事情**。Φ5 是这个连续性的物理载体。同时它还规定了"分层"——全局配置和项目本地配置分开存，避免一个项目的配置污染另一个项目。

**反例假设：** 若不成立 → 配置散落、每次重新询问、运行结果不稳定。

**反例假设强度：** 强

**覆盖：** 10+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P012 | 配置就地分层（全局 ~ vs 项目本地） | clean-git, release-project, sync-agent, contribute-skill |
| P045 | profile 是 Execute 阶段唯一依据 | release-project, clean-git, sync-agent |
| P056 | state.json 与 config.json 分离 | sync-agent |
| P122 | 上下文"通用 → 项目特定"叠加 | sync-design |
| P144 | 捕获目标由配置决定，与 cwd 无关 | capture-insight |

**判断方法：** 这份配置如果换个项目还有意义吗？有意义 → 全局；无意义 → 项目本地。

**反例边界：** 不是"所有数据都持久化"——一次性的对话上下文、临时计算结果不该写盘。Φ5 只管**跨会话需要的配置**。

---

## Φ6 单一真理源 + 动态读取

**章旨：** Bundle 列表、规范、注册表、版本号等从 SoT 文件动态读取，不在 SKILL.md 中硬编码。

**价值：** Skill 文档容易过时——CLAUDE.md 写"有 4 个 bundle"，但 skills-index.json 已经加到 7 个。**硬编码会让规范漂移成为常态。** Φ6 通过"运行时读取"保证 skill 永远与系统真实状态同步。

**反例假设：** 若不成立 → 规范漂移、列表过时、需手动同步 N 处。

**反例假设强度：** 中

**覆盖：** 8+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P022 | 动态读取目标系统的 SoT，不硬编码 | contribute-skill, init-skill, publish-skill, archive-skill |
| P033 | 用 Read 工具动态加载规范文件 | init-skill |
| P114 | 实际计数列清单，禁止估算 | survey-skillrepo |
| P117 | 字段名→Token 映射表作为权威依据 | build-style |
| P129 | 操作前必须读目录方法论 frontmatter | manage-dir, manage-docs |
| P074 | Reference 文件索引表 | dispatch-task, init-workflow, setup-debug |

**反例边界：** 不是"所有数据动态读"——skill 内部固定的步骤序列、固定的字符串模板可以硬编码。Φ6 只管**会随系统状态变化的数据**。

---

## Φ13 Skill 自我进化（积累写回 references/）

**章旨：** 执行中积累的新知识写回 references/ 或配置文件，供下次复用，让 skill 越用越省力。

**价值：** Skill 不是静态规范——每次执行可能学到新东西（一种新技术栈的日志方式、一个项目特有的发布流程、一类新的边界情况）。**如果不写回，每次会话都从零开始**。Φ13 让 skill 成为"积累型"工具而非"无状态"工具。

**反例假设：** 若不成立 → 每次会话重新踩坑，知识不积累。

**反例假设强度：** 中

**覆盖：** 4 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P091 | 新发现写回 references/tech-stacks/ | setup-debug |
| P013 | 配置缺失用默认 + 完成时询问生成 | clean-git, release-project, contribute-skill |
| P123 | manifest 演进，发现新区域时提示扩展 | sync-design |
| P132 | spec 处理完后加 frontmatter 闭环元数据 | migrate-spec |

**反例边界：** 不是"所有发现都写回"。setup-debug 给出明确五条准入标准（可复用 + 非显然 + 影响实质 + 技术栈归属 + 未被覆盖），全满足才写。

# 三组｜用户交互

四条哲学共同回答：**与用户沟通时的颗粒度和形式。**

## Φ4 信息按需收集

**章旨：** 上下文推断 → 显式指定 → 扫描列出（三级 fallback）；能推断的不问，不能推断的逐项澄清。

**价值：** Skill 启动时需要参数，但**问太多会赶走用户，问太少会出错**。Φ4 是这两个极端之间的工程平衡。"按需"的核心是**主动用上下文降低询问数量**，仅在真的不可推断时才请用户。

**反例假设：** 若不成立 → 要么逼用户提供一切，要么瞎猜。

**反例假设强度：** 中

**覆盖：** 11+ skill（覆盖最广的哲学之一）

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P020 | 三级 fallback 链：上下文 → 显式 → 扫描 | contribute-skill, init-skill, publish-skill |
| P032 | Step 0 需求澄清每次只问一个 | init-skill |
| P113 | 按文件层级逐级搜索设计意图 | survey-skillrepo |
| P112 | 上下文已含模式信号则跳过询问 | learn-video, capture-todo |
| P138 | PDF 读取分段且检测到 References 自动停 | learn-paper |
| P128 | references/ 按需读取，按图类型/场景区分 | draw-diagram |
| P134 | 提供"仅标注模式"跳过完整流程 | migrate-spec |
| P145 | 三信号推断项目，唯一高置信才直接确认 | capture-todo |
| P111 | 用 AskUserQuestion 让用户主动选择 | learn-video, forge-doc |

**反例边界：** 不是"完全不问用户"——歧义性强的场景仍需用户决策。Φ4 只是**抬高询问门槛**，不是消灭询问。

---

## Φ7 错误可分级

**章旨：** 区分阻塞错误和建议警告，警告不阻止 skill 通过。

**价值：** "全部正确 vs 全部失败"二元判断太粗——很多场景下警告值得用户看到但不该挡住流程。Φ7 让 skill 能输出"通过但有改进建议"这样的中间状态。

**反例假设：** 若不成立 → 所有问题等同对待，要么过严要么过宽。

**反例假设强度：** 中

**覆盖：** 6 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P040 | 警告与错误分级，警告不阻止通过 | publish-skill, init-skill, clean-git |
| P035 | ✓/! 分级提示符号 | init-skill, publish-skill, clean-git |
| P017 | 分组展示（明显可删 / 可能可删 / 保留） | clean-git, publish-skill |
| P104 | 退出码 2 表示"无内容"触发降级 | extract-vision |

**反例边界：** Φ7 不是"什么都允许通过"——结构性错误仍是阻塞。Φ7 划分的是**严重度梯度**，需要为不同梯度设计不同响应。

## Φ10 输出符号统一

**章旨：** ✓ ! ✗ ━━ 使用统一符号集；状态标签用大写英文（UPDATED/NEW/UNCHANGED）。

**价值：** Skill 输出大量结构化信息（状态、操作清单、报告）。**如果每个 skill 自创视觉语言，用户每次都要重新学习**。Φ10 是跨 skill 的视觉一致性约定，本质是认知效率优化。

**反例假设：** 若不成立 → 用户每次重新学符号含义，沟通成本累积。

**反例假设强度：** 中

**覆盖：** 7 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P010 | 完成后输出 ✓ 摘要 + 文件路径 | archive-skill, contribute-skill, init-skill, publish-skill, clean-git |
| P079 | 差量报告 UPDATED/UNCHANGED/NEW | init-workflow, publish-skill, contribute-skill |
| P083 | 表格展示文件操作状态 | init-workflow, publish-skill, contribute-skill |
| P127 | 明确禁用 Unicode + ASCII 替换表 | draw-diagram |
| P030 | 报告结构模板内置 | dedup-skill, publish-skill, clean-git |

**反例边界：** Φ10 不是文字风格统一，而是**机器/人共读的状态符号统一**。skill 之间正文风格可以不同，但状态符号必须一致。

## Φ14 一次一问，保留用户原始颗粒度

**章旨：** 澄清模糊意图时一次只问一个，让用户主导节奏；不替用户把模糊表达升格为确定性判断。

**价值：** Skill 容易陷入"机器思维"——一次性抛出问题列表让用户"汇报"。但用户的思维是迭代式的：回答第一个问题时往往会引出对第二个问题的修正。Φ14 是对**用户认知节奏**的尊重，区别于 Φ4 的"减少询问"，Φ14 管的是"如何问"。

**反例假设：** 若不成立 → 捕获用户表达的 skill 会一次性列出多个问题、会主动升格用户的模糊判断。

**反例假设强度：** 中

**覆盖：** 3 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P140 | 一次只问一个，等用户答完再问下一个 | capture-insight, capture-todo |
| P141 | 保留用户的不确定语气，不升格为强声明 | capture-insight |
| P142 | 澄清前先用一句话复述核心让用户校准 | capture-insight |
| P032 | Step 0 需求澄清每次只问一个 | init-skill |

**反例边界：** Φ14 不否定批量决策——dispatch-task 的判断决策树和 clean-git 的分组确认都是合理的批量场景。Φ14 适用于**模糊意图的澄清阶段**，那时用户自己也不知道想要什么。

**与 Φ1 的边界张力：** Φ14（一次一问）与 Φ1（一次性摘要 + 一次确认）看似冲突——前者拆开问，后者合并问。实际上它们对应不同阶段：**Φ14 在输入未明阶段，Φ1 在输入已明、即将执行阶段。**

# 四组｜结构与组合

四条哲学共同回答：**skill 的结构、与其他 skill 的关系、与任务的关系。**

## Φ3 边界显式且穷举

**章旨：** 行为边界（"不在范围内"节）+ 失败边界（"边界情况表"）都明确写出。

**价值：** Skill 在对话流中容易被无限扩展——用户问"顺便帮我也做 X 吧"，模型容易答应。**没有显式边界，skill 会逐渐变成什么都做的混合体。** 同时失败路径也需要穷举，否则遇到边界对话时只能临时决定，导致行为不一致。

**反例假设：** 若不成立 → skill 被对话扩展，边界对话时临时决定。

**反例假设强度：** 中

**覆盖：** 8+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P009 | 设"不在范围内"节明确边界 | archive-skill, contribute-skill, dedup-skill, init-skill |
| P008 | 文件末尾设边界情况汇总表 | archive-skill, contribute-skill, dedup-skill |
| P125 | 显式列出错误选型反例 | draw-diagram |
| P073 | 负面示例 + 正面示例对比 | dispatch-task |
| P109 | 不顺着用户意图临时改变 skill 职责 | learn-skill |
| P072 | 错误按类型分类处理 | dispatch-task, init-workflow |
| P041 | 报告按问题类型分类展示 | publish-skill, clean-git, archive-skill |

**反例边界：** Φ3 不是"列出所有可能情况"——那不可能。是"列出常见、易踩坑的情况"。Φ3 的价值在于**让边界讨论从隐式变成显式**。

## Φ8 Skill 可组合，不孤立

**章旨：** 通过"下一步"交棒、外部生态、目录协作形成可组合生态。

**价值：** 单个 skill 是单点工具，组合起来才是生态系统。**Φ8 是 skill 系统从"一组工具"升级为"工作流"的关键。** 它要求 skill 不只完成自己，还要主动指向下一步——这与"做完就退出"的 Unix 工具哲学不同。

**反例假设：** 若不成立 → skill 孤岛运行，用户不知如何串联。

**反例假设强度：** 中

**覆盖：** 11+ skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P011 | 完成后给"下一步"交棒提示 | archive-skill, contribute-skill, init-skill |
| P052 | 复用外部生态原生机制 | runby-opencode |
| P053 | 一次配置 skills.paths 多 skill 自动可用 | runby-opencode |
| P060 | 与生态约定自动维护 .gitignore→.stignore | sync-agent |
| P116 | 调查与配置生成由两个独立 skill 完成 | scout-brand → build-style |
| P105 | 广扫与深读拆为独立 skill 交棒 | survey-skillrepo → learn-skill |
| P147 | 为 hook 警告预先给出确认指引 | capture-todo |

**反例边界：** Φ8 不要求"所有 skill 都要交棒"——独立工具型 skill 可以无下游。Φ8 是**有自然下游时不要遗漏**。

---

## Φ11 关注点分离（内部分层）

**章旨：** 内部分层明确，组件间不混合职责——消息层 vs 数据层、capture 层 vs 应用层、MANAGED 块 vs 用户代码。

**价值：** 区别于 Φ9（对外不越权），Φ11 管的是**skill 内部多层架构的边界**。一个 skill 内部往往有多个角色（捕获器、解析器、写入器），如果它们职责混淆，会出现"捕获层修改了原始数据""消息层包含全部业务细节"这类反模式。

**反例假设：** 若不成立 → 抽象泄漏，组件 A 改了 B 的状态，调试时难定位。

**反例假设强度：** 中

**覆盖：** 6 skill

**下属范式：**

| 编号 | 范式 | 来源 skill |
|------|------|-----------|
| P071 | 消息层简洁，数据层详细 | dispatch-task |
| P082 | MANAGED 块外用户代码永不覆盖 | init-workflow |
| P096 | capture 层不修改原始内容 | setup-debug |
| P098 | 不改变应用行为（捕获层透明） | setup-debug |
| P110 | rerun 用 DAG 内部名而非进度显示名 | learn-video |
| P115 | 知识层与映射层分离 | build-style |
| P143 | 笔记类提交走永久 chore 分支不污染功能 diff（原 Φ17） | capture-insight, capture-todo |

**与 Φ9 的区别：**
- **Φ9**：对外部不越权（不动主干、不动远端、不动用户已有文件）
- **Φ11**：对内部不越权（捕获不改原数据、消息不抢数据职责、MANAGED 不动用户区）

## Φ15 任务分阶段，读完再写

**章旨：** 复杂任务拆成多个阶段串联执行，每个阶段职责单一——I/O 与 LLM 分开、广扫与深读分开、读全部与写一份分开。

**价值：** 单个 agent 一气呵成处理复杂任务有三个问题：(1) context 溢出 (2) I/O 等待时 LLM 闲置 (3) 边读边写产生重复或矛盾。Φ15 通过**强制时序拆分**解决这些。

**反例假设：** 若不成立 → 跨阶段任务用单 agent 一气呵成，会丢维度（context 溢出 / LLM-IO 不可并行 / 边读边写产生重复）。

**反例假设强度：** 强

**覆盖：** 5+ skill

**这条哲学有多种形态，但反例相同：**

**形态一：I/O vs LLM 拆分**
| 范式 | 来源 skill |
|------|-----------|
| P103 拆 LLM 密集 + I/O 密集为独立 subagent | extract-url, extract-vision |

**形态二：广扫 vs 深读**
| 范式 | 来源 skill |
|------|-----------|
| P105 仓库级与单元级拆为独立 skill | survey-skillrepo → learn-skill |
| P116 调查与配置生成拆分 | scout-brand → build-style |

**形态三：多遍读**
| 范式 | 来源 skill |
|------|-----------|
| P135 三遍读 PDF（定向 → 提取 → 批判） | learn-paper |

**形态四：读全部再写**
| 范式 | 来源 skill |
|------|-----------|
| P131 跨文档迁移先全部读完再做横向矩阵 | migrate-spec |

**反例边界：** Φ15 不要求**所有任务都拆阶段**——简单任务一遍完成更高效。Φ15 适用于**已知会超 context 或需要全局视野的任务**。

# 五组｜形式与表达约定

两条哲学共同回答：**SKILL.md 长什么样、运行时输出长什么样——形式契约的两个时态。**

注：本组哲学性质特殊——它们位于哲学层级的**层 B 约定层**，介于"层 A 系统机制"和"层 C 设计哲学"之间。违反不致命（不像 Φ12 注入会出安全事故），但累积成本明显（认知、路由、维护）。

## Φ18 形式即接口（静态文档约定）

**章旨：** SKILL.md 不是实现文档，而是接口规范——在 Skill 系统机制（层 A）已规定的硬性形式之上，约定层的形式选择（命名风格、描述格式、节顺序、文件组织）影响读者认知效率和 LLM 路由准确率。

**价值：** Skill 机制只规定最低形式（必须有 frontmatter、name 与目录名一致、description 非空）。但这些机制最低要求之上，仍有大量设计选择——name 用什么格式？description 怎么组织触发词？节顺序怎么排？这些选择本身是契约：所有 skill 共享同一套约定，读者和 LLM 才能可预测地工作。

**反例假设：** 若不成立 → 每个 skill 形式各异，读者每次重新学习结构，LLM 路由因 description 格式漂移而准确率下降，维护成本累积。

**反例假设强度：** 中

**覆盖：** ~30 skill（隐式收敛，无人显式偏离）

**与 Skill 机制的关系（重要）：**

| 项 | 性质 | 归属 |
|----|------|------|
| name 与目录名一致 | Claude Code 加载器硬性要求 | **层 A 机制** |
| frontmatter 字段必填 | 加载器解析必需 | **层 A 机制** |
| description 必须存在非空 | LLM 路由必需 | **层 A 机制** |
| 注册到 skills-index.json | publish-skill 工具要求 | **层 A 机制** |
| name 格式 `<verb>-<noun>` 词表 | 机制允许任意合法字符，词表是社区选择 | **层 B 约定（Φ18）** |
| description 用 Triggers 格式 | 机制允许自由文本，结构是约定 | **层 B 约定（Φ18）** |
| 节顺序固定 | LLM 能读任何顺序，约定提高可预测性 | **层 B 约定（Φ18）** |
| Step 命名"动词+名词" | 纯风格 | **层 B 约定（Φ18）** |
| references/ 提取阈值 | 机制不限制文件数量 | **层 B 约定（Φ18）** |

**Φ18 只管"约定"那一层，不管"机制"那一层。**

**下属范式：**

| 编号 | 范式 | 等级 | 来源 |
|------|------|------|------|
| G1b | name 用 `<verb>-<noun>`，动词在标准词表 | SHOULD | 隐式收敛，30 skill |
| G5 | description 格式 `<动词短语>. Triggers: '...'` | SHOULD | 隐式收敛 |
| G6 | description 不与其他 skill 触发词重叠 | SHOULD | 反例：路由模糊 |
| G7 | 节顺序：触发条件 → 步骤 → 不在范围内 | SHOULD | 隐式收敛 |
| G9 | Step 命名"动词 + 名词" | SHOULD | 隐式收敛 |
| G12 | references/ 超过 20 行的内容提取到子文件 | SHOULD | 反例：单文件膨胀淹没主流程 |
| G13 | 平台特定内容分层放 references/ | MAY | 扩展模式 |

**反例边界：** Φ18 不管"内容质量"、不管"运行时行为"、不管"代码风格"。只管 SKILL.md 静态形式的可预测性。

**研究亮点：** Φ18 是在三向对照中发现的——这条哲学之前藏在"元规则"里，被混入"系统机制"。研究方法本身（三向对照 + 机制 vs 约定区分）暴露了它。这证明哲学体系的发现是迭代过程，不是一次到位。
