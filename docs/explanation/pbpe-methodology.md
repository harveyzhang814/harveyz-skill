# 范式驱动的设计哲学提取法（PBPE）

> Pattern-Based Philosophy Extraction

从一组已存在的同类制品中，归纳出**支配其设计的哲学**，并衍生**规范性规则**的方法论。本次 skill 哲学研究的副产品。

---

## 一、方法定位

**用途：** 给同质制品（skill、API、文档、组件库）建立规范，但既不想凭空立论，也不想只描述现状。PBPE 用归纳产生哲学、用反例验证哲学、用哲学衍生规则，形成**归纳 → 演绎**的闭环。

**与三类传统方法的差异：**

| 方法 | 自顶向下立论 | 自底向上归纳 | 反例验证 | 等级规范化 |
|------|-----------|------------|---------|-----------|
| Grounded Theory | ✗ | ✓ | ✗ | ✗ |
| Design Guidelines | ✓ | ✗ | ✗ | ✓ |
| Philosophy 文章 | ✓ | ✗ | 弱 | ✗ |
| **PBPE** | ✗ | ✓ | ✓ | ✓ |

---

## 二、核心概念区分

PBPE 的清晰前提是区分四层概念：

### 2.1 范式 vs 哲学 vs 规则

```
       范式（pattern）         直接观察："X 做了 Y"
        ↓ 聚类 + 反推
       哲学（philosophy）       推论意图："因为我们相信 Z"
        ↓ 演绎
       规则（rule）             规范输出："应该做 Y"
```

| 层次 | 形态 | 来源 | 数量级 |
|------|------|------|--------|
| 范式 | 一句做什么 + 一句反例 | 直接观察 | 每样本 5–10 条 |
| 哲学 | 章旨 + 反例假设 | 聚类反推 | 10–20 条 |
| 规则 | MUST / SHOULD / MAY | 哲学演绎 | 哲学数 × 3–5 |

### 2.2 层级区分（关键陷阱）

不是所有"必须遵守的事"都是设计哲学。须区分四层：

| 层 | 性质 | 违反后果 | 例子（来自 skill 系统） |
|----|------|---------|---------------------|
| A 系统机制 | 系统硬性规定 | 系统拒绝 | name 与目录名一致、frontmatter 字段 |
| B 约定层 | 系统允许但社区收敛 | 工作但次优 | name 词表、节顺序、视觉符号 |
| C 设计哲学 | 真正的设计选择 | 错误或风险 | 故障保留、不越权、注入防御 |
| D 团队约定 | 仅本团队选择 | 团队不一致 | 正文用中文 |

**PBPE 只处理层 B 和 C，明确剥离 A 和 D。**

> 判别方法：问"违反这条会怎样"？
> - 系统直接拒绝 / 报错 → 层 A
> - 系统接受但工作不畅 → 层 B
> - 系统接受但有副作用或风险 → 层 C
> - 系统无影响，只是团队偏好 → 层 D

这是最容易混淆的方法论陷阱——把系统 spec 误判为设计哲学会让规范臃肿、可信度下降。

## 三、完整流程

### Phase 0 — 封存预期哲学（对冲选择偏差）

**目的：** 研究者已有先验，必须封存以便事后量化。

**操作：**
1. 在读任何样本之前，写下"我预期会发现什么哲学"（5 条左右）
2. 封存到固定文件（commit）
3. 研究结束时与实际产出对比，**未命中项必须主动反向搜索**

**反向搜索机制：** 若 Phase 0 某条预期完全未观察到，必须明确判定：
- 是研究者漏看 → 补观察
- 是范围所限（样本无对应场景）→ 标"待扩展验证"
- 是先验错误 → 在自评中明确披露

这把 Phase 0 从"事后审计"升级为"主动发现机制"。

---

### Phase 1 — 批量范式抽取（严格粒度）

**目的：** 仅观察现象，不做高阶推断。

**操作：**
1. 完整读取每个样本
2. 按顺序抽取范式，编号 P001、P002…
3. 每条范式必须满足三个字段：

| 字段 | 要求 |
|------|------|
| 做什么 | 一句话描述具体行为 |
| 反例 | 一句话描述违反时的具体行为 |
| 出现样本 | 列出所有展现该范式的样本 |

**粒度规则（严格）：**
- 一个范式 = 一句"做什么" + 一句"不做什么"
- 描述含 "且" / "以及" / "+" / "包含" → 必须拆分
- 给不出明确反例 → 不是范式，是惯例，丢弃

**关键纪律：** Phase 1 不做匹配、不聚类、不反推哲学。这是最重要的纪律——**边读边匹配会造成顺序依赖**，先读的样本锚定后续命名空间，最终结果取决于读取顺序而非内容本身。

**中间产物：** 范式登记表（CSV-like），便于 Phase 2 操作。

---

### Phase 2 — 离线聚类 + 反推哲学

**2a. 聚类：** 读完全部样本后，对全部范式做一次性聚类。多轮迭代直到稳定。

**2b. 反推哲学：** 每个聚类反推一条哲学，**必填**以下字段：

```yaml
哲学编号: Φ-N
章旨: <动词短语 + 对比项> 或 <单一短语>
覆盖样本数: N
下属范式: [P001, P003, ...]
反例假设: 若此哲学为假，样本会变成...
观察到的反例样本: [列表]
反例假设强度: 强 / 中 / 弱
```

**反例假设强度判定：**
- **强**：反例描述的失败明显且不可接受（数据丢失、安全事故、协作故障）
- **中**：失败可恢复但成本明显（认知累积、维护负担、调试困难）
- **弱**：仅影响美观或便利性

**章旨命名格式（统一）：**
- 格式 A：`<动词短语>，优于<对比项>`（如 "可回退，优于不可回退"）
- 格式 B：单短语（如 "故障保留现场"）
- 全文统一选其一

---

### Phase 3 — 仲裁 + 阈值过滤

#### 3.1 仲裁规则（三指标可计算）

冲突哲学之间用以下指标判定胜负：

| 指标 | 定义 | 用途 |
|------|------|------|
| 覆盖数 | 下属范式覆盖的去重样本数 | 越高越胜 |
| 反例数 | 观察到的反例样本数 | ≥2 则哲学失效 |
| 反例假设强度 | 强 / 中 / 弱 | 等强度时辅助决策 |

#### 3.2 阈值过滤（双门槛）

| 类型 | 强候选 | 弱候选 | 不纳入 |
|------|--------|--------|--------|
| 范式 | ≥ 3 样本 | 2 样本 | 1 样本 |
| 哲学 | 解释 ≥ 3 范式 + 强度 ≥ 中 + 反例 ≤ 1 | 不满足但有意义 | — |

#### 3.3 排除规则

- 普及度 ≥ 80% → 工程惯例（不算设计选择），剔除
- 反例样本 ≥ 2 → 哲学失效
- 反例强度 = 弱 → 不通过

---

### Phase 4 — 衍生规则 + 自评

#### 4.1 等级映射（客观规则）

```
覆盖率 ≥ 60% → MUST
覆盖率 30–60% → SHOULD
覆盖率 < 30% → MAY

破坏性修正（+1 级）：
  规则涉及破坏性操作（删除 / 远端 / 覆盖 / 注入）→ 等级提升一级
  例：MAY + 破坏性 → SHOULD
```

#### 4.2 规则标注两类来源

- **[观察]** ≥ 3 样本已实现该规则
- **[衍生]** 哲学逻辑要求但样本实现不足——这是哲学体系**前瞻性**的体现

**衍生规则的价值高于观察规则**——它们是"哲学要求但还没人做到"的部分，往往是规范升级的核心机会。

#### 4.3 必填自评字段

- 与 Phase 0 封存清单的命中率
- 未命中项的反向搜索结论
- 仍存在的偏差源（单研究者、样本偏窄、强度判断主观等）

---

## 四、关键技术细节

### 4.1 反例测试（防单点幻觉）

每条哲学必须通过：

> "如果这条哲学为假，样本会变成什么样？"

无法回答 → 哲学反推错误，丢弃。

**这是 PBPE 最重要的硬性约束。** 没有反例测试，反推哲学是无约束的修辞游戏。同一范式可被反推成 3 条完全不同的哲学，取决于研究者当下措辞偏好。

### 4.2 仲裁的可计算性

早期版本用"覆盖数 / 解释残差 / 正交性"作为仲裁指标，但后两者无法在中等样本规模下操作（< 30 样本）。

改为"覆盖数 / 反例数 / 反例假设强度"——三者都可在 10 样本规模下计算。

### 4.3 跨域验证（cross-bundle）

单批样本会有特性偏差。PBPE 要求**至少 2 个独立子域验证**：

- 单子域：可能漏掉子域特殊性的哲学
- 多子域：跨子域不稳定的哲学暴露

**实证：** 本次 skill 研究在 meta bundle 单域时只发现 10 条哲学，扩展到 coding 后浮现 Φ11，扩展到 research/experiment 后浮现 Φ12 强样本。**没有跨域，3 条核心哲学会被漏掉。**

### 4.4 Subagent 委派（处理规模）

样本量 > 15 时，主对话 context 不够。委派给 subagent 并要求严格遵守 Phase 1 抽取纪律——**主对话保留聚类与判定权，subagent 只做范式提取**。

### 4.5 反向搜索（Phase 0 未命中追查）

每条 Phase 0 预期未命中时**主动反向搜索**：
- 抽样所有样本核对预期场景是否真不存在
- 不存在 → 标"性质所致，待扩展验证"
- 漏看 → 补观察后重跑

**实证：** 本次研究 Phase 0 预期"变量隔离"在 meta + coding 两个 bundle 未观察到，反向搜索确认"meta bundle 变量都是受控值"——这让研究者从"是否漏看"的反复焦虑中解脱出来，并产生具体下一步行动（扩展到 research bundle 找样本）。

## 五、偏差对冲机制总结

PBPE 的偏差源与对冲：

| 偏差 | 对冲机制 |
|------|---------|
| 顺序依赖（先读样本锚定命名空间） | Phase 1 不反推，Phase 2 离线聚类 |
| 反推哲学的单点幻觉 | 反例测试 + 反例样本搜索 |
| 仲裁规则模糊 | 三指标可计算化 |
| 范式准入过严丢失弱信号 | 双门槛（强 ≥3 + 弱 2） |
| 选择偏差（只看到想看的） | Phase 0 封存 + Phase 4 自评 + 反向搜索 |
| 单 agent 主观 | 接受局限，明确披露 |
| 规模膨胀难管理 | 中间产物（范式登记表） |
| 机制 vs 设计混淆 | 四层结构（机制 / 约定 / 设计 / 团队） |

## 六、输出形态

PBPE 标准产物三件套：

### 6.1 范式登记表（Phase 1 产出）

CSV-like 表格，便于检索和重新聚类：

| 编号 | 做什么 | 反例 | 出现样本 | 来源行号 |
|------|--------|------|----------|---------|
| P001 | 操作前展示完整摘要等待 y/n 确认 | 直接执行无确认 | A, B, C, D, E | A#65, B#34 |

### 6.2 哲学知识文档（Phase 2 产出）

每条哲学含 7 字段（章旨 / 价值 / 反例假设 / 强度 / 覆盖 / 下属范式 / 反例样本），加上哲学间张力点和演化日志。

### 6.3 衍生规则文档（Phase 4 产出）

规则按等级 MUST / SHOULD / MAY 分组，每条标 [观察] / [衍生]，加上与现有规范的差异分析。

### 6.4（可选）对照报告

若已存在规范，做三向对照：
- A 重叠区
- B 现有规范有 / 哲学缺
- C 哲学有 / 现有规范缺 ← 重点

## 七、方法学血统

PBPE 借鉴：

| 主流方法 | 借鉴点 |
|---------|--------|
| Grounded Theory（社会学） | 自底向上归纳 + 编码 |
| ESLint Rule Categories | 维度分类 + 严重度分级 |
| Google Style Guide | 哲学前言 + 规则正文分离 |
| RFC 2119 | MUST / SHOULD / MAY 等级语言 |
| OWASP ASVS | 多轴矩阵分类（验证等级 + 威胁域）|
| TRIZ（创新方法学） | 反例驱动验证 |

**PBPE 的原创贡献：**
1. **四层结构**（机制 / 约定 / 设计 / 团队）—— 防止把系统 spec 误判为设计选择
2. **反例必填字段** —— 把口头反例测试硬性化
3. **离线聚类 + Phase 0 反向搜索** —— 双重对冲选择偏差
4. **衍生规则 vs 观察规则区分** —— 让哲学体系的前瞻性可量化

## 八、适用与不适用边界

### 适用场景

| 场景 | 是否适用 |
|------|---------|
| 同质制品归纳（skill / API / 组件库 / 文档模板） | ✓ 强适用 |
| 跨领域比较（skill vs API） | ✗ 同质前提缺失 |
| 单制品深度分析 | ✗ 样本太小 |
| 已有完整规范要审计 | ✓ 用三向对照 |
| 完全空白领域要立规范 | ✓ 但需 ≥ 10 样本 |

### 适用门槛

- **样本数 ≥ 3**：低于此无法过准入门槛
- **理想样本数 10–30**：能产 10–20 哲学
- **单 agent 上限约 30 样本**：超此规模需 subagent 并发

### 已知局限

- **单研究者主观**：无 inter-rater agreement，承认局限
- **反例假设强度仍主观**：强/中/弱三档无客观锚点
- **演化滞后**：哲学列表会随新样本变化，需周期性重做
- **样本同质性假设**：跨子域差异太大时聚类会失败

## 九、半年后重做指南

PBPE 产物会过期。建议演化策略：

| 触发条件 | 操作 |
|---------|------|
| 加 5 个新样本 | 局部更新——看新样本是否产生新范式 |
| 加 15 个新样本 | 重做 Phase 2 聚类（不重抽 Phase 1） |
| 加 30 个新样本 | 完整重跑（重新封存 Phase 0） |

演化日志记录每次重做的产物变化（新哲学、折叠、弃用）。

## 一句话总结

> **PBPE 是用反例倒逼归纳、用层级剥离机制、用阈值过滤偏差、用衍生检验完备性的归纳—演绎闭环方法论。**

适用于任何同质制品的规范化任务，下次想给"一组类似的东西"立规范时都可复用。
