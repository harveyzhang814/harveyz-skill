# question-me v2：动态决策树设计

**日期：** 2026-07-01  
**状态：** 待实现  
**前置文档：** `2026-07-01-question-me-design.md`（v1 基础设计）

---

## 问题

v1 的决策树在 Phase 1 三问全答完后**一次性生成**，之后树结构固定不变。这导致两个缺陷：

1. **G/S 缺少子节点**：三问全答完时，Claude 已有 S 和 SC 的上下文，导致 G 的子问题被"上下文消化"而跳过——但如果单独只看 G 的答案，G 下面可能需要追问
2. **树不会增长**：Phase 2 里用户的回答无法创造新节点，即使答案明显打开了新方向

---

## 核心设计变更

### 变更 1：Phase 1 改为"问→答→生成子节点"循环

**旧流程：**
```
Q1 → Q2 → Q3 → 全部答完 → 一次性初始化决策树
```

**新流程：**
```
Q1 → 用户答 → [必须] 评估子节点 → 追加 → 渲染（--open）
Q2 → 用户答 → [必须] 评估子节点 → 追加 → 渲染
Q3 → 用户答 → [必须] 评估子节点 → 追加 → 渲染
→ 进入 Phase 2
```

每个 Phase 1 节点答完后立即评估：这个答案是否引出了新的不确定性？若是，生成子节点并加入树。**评估是强制步骤，不可跳过。**

### 变更 2：Phase 2 每轮追加子节点评估 + 兄弟扫描

**旧流程：**
```
选最高优先级 open 节点 → 问 → 更新 done → 渲染 → 继续
```

**新流程：**
```
选最高优先级 open 节点 → 问 → 用户答 → 更新 done
  → [必须] 子节点评估：生成 0-N 个子节点
  → [必须] 兄弟扫描：此答案是否让同级兄弟节点变无关？→ 标 skip
  → 渲染 → 继续
```

---

## BFS 生成规则

每次生成子节点时，遵循以下规则：

**生成时机：** 节点变为 `done` 的那一轮，立即生成，不延迟

**生成数量：** 一次生成所有潜在子节点（0 到 N 个），不逐个生成

**生成上限：** 每个父节点最多 1 层子节点（不预生成孙节点）；孙节点等到子节点答完时再评估

**生成标准：** 子节点只在"答案存在真实不确定性且影响执行方向"时生成。以下情况不生成：
- 答案已足够明确，无需追问
- 子问题 Claude 可以合理默认处理（直接标为 `infer`）
- 子问题的答案不影响实现方向

**标记为 skip：** 已生成的子节点，若兄弟扫描发现它因其他答案变得无关，标为 `skip`，不再追问

## 分支停止与叶子节点

**分支停止是局部的，不是全局的。**

每个节点独立决定自己是否需要子节点。"不生成子节点"就是分支到此停止的结果——不需要额外的状态标记：

| 情况 | 节点状态 | 子节点 | 含义 |
|------|----------|--------|------|
| 答完，有追问价值 | `done` | 有子节点 | 分支继续 |
| 答完，无追问价值 | `done` | 无子节点 | 分支自然结束（叶子） |
| Claude 可默认处理 | `infer` | 无子节点 | 叶子，默认假设在摘要透明列出 |
| 兄弟答案让它无关 | `skip` | 无子节点 | 叶子，跳过 |

**叶子节点不是一个状态，是结构属性**——看有没有子节点就知道了。`done` 无子节点即为叶子。

**"切换分支"不是主动操作。** 某分支的 open 节点耗尽后，优先级算法自动把下一个可选节点提上来，效果就是切换。不存在"暂停某分支、稍后回来"的机制。

**全局停止是自动的。** 当所有 open 节点变为 done / infer / skip，树自动清空，进入 Step 4。Claude 不需要主动判断"该停了"。

## 兄弟扫描（局部扫描）

答完节点 X 后，只扫描 **X 的同级兄弟节点**：

> 这个答案是否让某个兄弟节点变得无关或矛盾？若是，标 `skip`。

不扫描跨分支的远端节点，原因：
- 一个节点的答案影响远端节点的概率极低
- 扫描范围过宽容易产生噪音（牵强的 skip 或新节点）
- 真正的跨分支影响（方向性修正）属于例外情况，单独处理

**例外：重大方向修正**

当用户答案与已答的祖先节点产生根本矛盾时（如 SC 已 done 为"只加脚本"，但用户后来说"其实也要改抓取逻辑"），不走兄弟扫描，而是：
1. 显式指出冲突
2. 询问用户是否需要修改已答节点
3. 用户确认后重新标记受影响节点为 `open`，重新提问

---

## 遍历规则（不变）

提问顺序与 v1 保持一致：

> 从所有 `dep` 已满足（或无 `dep`）的 `open` 节点中，选被其他节点依赖次数最多的（影响面最大）

这个规则自然产生：
- **DFS 行为**：子节点因依赖父节点而高优先级，倾向于先挖完一条链
- **BFS 行为**：当一个分支的子节点影响面不如同级兄弟时，自动切换

不需要硬编码 DFS 或 BFS，影响面排序已是最优策略。

## 树生长示意

```
Phase 1 — 问 Q1（目标）：
用户答：candidate_tags 导出为报告
[必须评估] 报告格式是否需要澄清？→ 是
→ 追加 [fmt:open] id=FT dep=G
树：
  [goal:done]  id=G
  [fmt:open]   id=FT  dep=G   报告格式

Phase 1 — 问 Q2（成功标准）：
用户答：能看频率和时间信息，手动决定升级
[必须评估] 还有不确定性吗？→ 否，答案足够具体
→ 不生成子节点（S 为叶子）
树：
  [goal:done]    id=G
  [fmt:open]     id=FT  dep=G
  [success:done] id=S

Phase 1 — 问 Q3（范围）：
用户答：只加导出脚本，不改现有抓取流程
[必须评估] → 输出路径来源不确定，notes 目录路径来源不确定
→ 追加两个子节点
树：
  [goal:done]    id=G
  [fmt:open]     id=FT  dep=G
  [success:done] id=S
  [scope:done]   id=SC
  [output:open]  id=OP  dep=SC   输出路径策略
  [source:open]  id=SR  dep=SC   notes 目录路径来源

Phase 2 — 选题：FT（1个下游dep=OP依赖它→1分），OP（被TR依赖→1分，dep=SC✓），SR（被DM依赖→1分，dep=SC✓）
→ 并列，选 FT（出现顺序靠前）

问 FT：报告格式？
用户答：文本文件即可
[必须评估] 无需子节点（够具体）
[兄弟扫描] FT 无兄弟
→ FT 成为叶子

Phase 2 — 选题：OP（1分），SR（1分）→ 选 OP

问 OP：输出路径策略？
用户答：只写文件，路径自动生成
[必须评估] 触发方式不确定 → 追加 [trigger:open] id=TR dep=OP
[兄弟扫描] SR 是 OP 的兄弟 → 此答案让 SR 变无关吗？→ 否
→ 树新增 TR

…以此类推，直到所有 open 节点清空 → 进入 Step 4
```

## 与 v1 的差异汇总

| 维度 | v1 | v2 |
|------|----|----|
| 树初始化时机 | Phase 1 三问全答后一次 | 每问答完即时追加 |
| G/S 是否有子节点 | 通常没有（被上下文消化） | 按需生成 |
| Phase 2 动态追加 | 不支持 | 每轮必须评估子节点 |
| 生成粒度 | 整树一次性生成 | 每节点答完生成该节点 1 层子节点 |
| 兄弟扫描 | 不支持 | 每轮扫同级兄弟 |
| 遍历顺序 | 影响面最大优先 | 不变 |
| 叶子节点标记 | 不适用 | 无单独状态，`done` 无子节点即为叶子 |
| 分支切换 | 不适用 | 优先级算法自动处理，无显式切换 |
| 全局停止 | open 节点清空 | 不变 |
| 重大方向修正 | 不支持 | 显式冲突提示 + 用户确认重标 |

---

## 不在范围内

- 跨会话持久化问答历史（每次独立）
- 子节点超过 1 层的预生成（孙节点动态生成）
- 自动执行（摘要确认后由 Claude 判断）
- 强制跑完所有节点（用户可随时打断）
- 跨分支远端节点的自动扫描（只扫兄弟）
