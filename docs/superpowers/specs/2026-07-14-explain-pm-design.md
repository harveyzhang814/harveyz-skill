# explain-pm Skill 设计

**日期：** 2026-07-14
**状态：** 已实现

---

## 背景

Agent 在讨论技术方案时容易堆砌技术术语（架构、实现细节、性能权衡），而遗漏"这对用户/产品意味着什么"的视角。explain-pm 是一个轻量的会话内 skill，把最近一段技术性表述从 PM 视角复述，并顺带指出其中值得从产品角度重新考虑的地方。

风格上参照 `rephrase`：单轮、简短、不追加流程负担。

---

## 触发

仅手动调用，不自动检测触发：

- `/explain-pm` — 无参数，复述调用前的最后一条 assistant 消息
- `/explain-pm <主题或文件路径>` — 复述/评估用户指定的内容（可以是一段议题描述，也可以指向具体文件）

---

## 执行

1. **取材料**：按上面的触发规则确定要复述的对象。
2. **转译**：把技术语言换成产品/业务语言——这段技术表述如果讲给 PM 听，PM 听到的是什么（用户能得到什么、解决了什么问题、有什么代价）。
3. **提出关注点（视情况）**：判断原表述从 PM 角度是否有值得重新审视的地方，比如：
   - 是否偏离了用户实际需求
   - 范围是否合理（做多了/做少了）
   - 是否过度工程化
   - 优先级是否搞错了
   如果技术表述本身已经很贴近产品目标、挑不出关注点，就只做转译，不硬造问题。
4. **输出**：在对话中直接回复，简短（几句话量级，不是长篇评审）。默认不写文件；仅当用户明确要求存档时，才把这段评论写入文件（写在哪个文件由用户当时指定，不预设固定路径）。

---

## 不做

- 不做多轮追问式澄清（那是 `question-me` 的职责）
- 不自动检测技术语言并主动插话
- 默认不生成/保存任何文档
- 不强制固定的输出小标题结构（"PM 听到的是什么/影响/关注点"这类模板），由 Claude 根据内容判断怎么组织这几句话最清楚
